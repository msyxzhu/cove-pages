LLM 是怎么变成 Agent 的

在聊 Agent 之前，我们先搞清楚一个根本问题：一个 LLM 本身不是 Agent。

LLM 只是一个"嘴"

你给 ChatGPT 发消息，它回复你——这只是一次函数调用：输入一段文字，输出一段文字。它不能打开文件、不能执行命令、不能记住上周的对话。LLM 就像一个嘴特别厉害但没有手、没有眼睛、没有记忆的大脑。

纯 LLM vs Agent

Agent = LLM + 环境交互能力

Agent 的本质就是：给 LLM 加上跟真实世界交互的能力。具体来说：

  • Tools（工具）— 让 LLM 能"动手"：执行代码、搜索网页、读写文件
  • Memory（记忆）— 让 LLM 能"记住"：跨会话保持上下文
  • Planning（规划）— 让 LLM 能"想清楚"：把大任务拆成小步骤
  • Loop（循环）— 让 LLM 能"持续干"：观察结果 → 调整 → 再行动

一句话总结：

核心公式
Agent = LLM + Tools + Memory + Planning + Loop
LLM 提供"思考"能力，其余部分让这个"思考"变成"行动"。

一个直觉类比

想象一个超级聪明但被关在黑屋子里的人。他啥都懂，但看不见外面、摸不到东西、也记不住昨天说了什么。Agent 开发就是给这个人装上眼睛、手、笔记本和行动清单，让他能真正做事。

常见误区
很多人以为"Agent = 更强的 ChatGPT"。不是的。Agent 的核心进步不在于模型更强，而在于系统设计——模型只是其中一个组件。同一个模型在不同的 Agent 系统里，表现可以天差地别。

ReAct 循环：思考-行动-观察

Agent 的核心运行机制叫 ReAct（Reasoning + Acting）。这是 2022 年由 Yao et al. 提出的论文，一举奠定了现代 Agent 的基本范式。

三步循环

ReAct 的核心就三步，不断循环直到任务完成：

  1. Reason（思考）— LLM 分析当前情况，决定下一步做什么
  2. Act（行动）— 调用一个工具去执行操作
  3. Observe（观察）— 拿到工具返回的结果，作为新的输入
ReAct 循环示意

一个具体例子

用户说："帮我查一下北京明天的天气，然后写一条适合发朋友圈的文案。"

// 第 1 轮
思考: 用户需要北京明天天气信息，我需要先搜索天气
行动: 调用 search_weather(city="北京", date="明天")
观察: 明天北京晴，22°C，微风

// 第 2 轮
思考: 拿到天气了，现在可以写文案了
行动: 直接生成文案（不需要工具）
输出: "明天北京 22°C 的阳光，适合出门被春风撩一下 🌸"

// 任务完成 ✅

为什么 ReAct 有效？

关键在于思考和行动的交错（interleaving）。之前的方法要么只推理不行动（Chain-of-Thought），要么只行动不推理（纯 tool-calling）。ReAct 让 LLM 每一步都先想清楚为什么要做这件事，再动手。

关键洞察
ReAct 循环就像一个医生看病：观察症状 → 思考可能的原因 → 开检查单 → 拿到结果 → 再思考 → 再开药。不是一口气下结论，而是边做边想、边想边做。
常见误区
ReAct 不是"每轮都必须调用工具"。LLM 完全可以在 Reason 阶段决定"不需要工具了，直接回答"。循环的关键是模型有选择权——它决定什么时候用工具、什么时候不用。

Tool Use：给 LLM 装上手脚

Tool Use（工具使用）是 Agent 能"做事"的基础。没有工具，LLM 只能说不能做。

工具是怎么传给 LLM 的？

你不是真的"安装"一个工具到 LLM 里。而是在 API 调用时，告诉模型有哪些工具可用，每个工具能做什么、接受什么参数。模型会在需要时"选择"调用某个工具。

Tool Use 工作流程

工具定义长什么样？

以 OpenAI 的 API 为例，一个搜索天气的工具定义：

{
  "type": "function",
  "function": {
    "name": "get_weather",
    "description": "获取指定城市的天气预报",
    "parameters": {
      "type": "object",
      "properties": {
        "city": {
          "type": "string",
          "description": "城市名称，如'北京'"
        }
      },
      "required": ["city"]
    }
  }
}

你传给 LLM 的就是这么一段 JSON。LLM 看到 description 就知道这个工具干嘛用。它返回的不是函数执行结果，而是 {"name": "get_weather", "arguments": {"city": "北京"}}——你拿到这个，自己去调真正的天气 API。

常见工具类型

类型例子典型用途
代码执行Python sandbox, shell数据分析、文件处理
信息检索Web search, RAG获取最新信息、查询知识库
文件操作read_file, write_file读写代码、文档
外部 API发邮件、创建 Issue与外部系统交互
浏览器截图、点击、导航UI 测试、网页操作
工具描述是"隐形 Prompt"
工具的 description 写得好不好，直接决定 LLM 会不会正确使用这个工具。这其实就是一种 Prompt Engineering——只不过不是写在对话里，而是写在工具定义里。

系统提示词：Agent 的"灵魂"

每个 Agent 都有一段 System Prompt（系统提示词）。这是 Agent 的"人格说明书"——告诉 LLM 它是谁、该怎么做、有什么规矩。

System Prompt 的作用

在每次 API 调用中，system prompt 排在最前面，优先级最高。它定义了：

  • 身份— 你是谁？（"你是一个 Python 开发专家"）
  • 行为规范— 该怎么做？（"先分析再动手"、"写代码前先跑测试"）
  • 边界— 不能做什么？（"不要修改生产数据库"）
  • 输出格式— 怎么回复？（"用中文回答，术语保留英文"）
消息在 API 调用中的结构

在 OpenClaw 中的实现

你正在跟我聊天，我的"灵魂"就是由这些文件组成的：

  • SOUL.md — 人格定义（"你是 Sage，把复杂变简单……"）
  • USER.md — 了解你是谁（轩，有编程经验，喜欢动手）
  • AGENTS.md — 行为规范（启动流程、记忆管理）
  • IDENTITY.md — 基础身份信息

这些文件在每次对话开始时被注入 system prompt，让我"记得"自己是谁。

System Prompt 就是 Agent 的 DNA
换一个 system prompt，同一个 LLM 立刻变成完全不同的 Agent。这也是为什么 OpenClaw 能同时运行 Sage（教学）、Dock（编码）、Helm（设计）——它们共享同一个模型，只是"灵魂"不同。
好的 System Prompt 的特征
具体 > 泛泛。"写代码要规范"不如"每个函数不超过 30 行，必须有类型注解"。
规则 > 祈求。"请不要……"不如"严禁……违反时停止并报告"。
示例 > 描述。给一个好输出的样本，比描述一百遍"怎么才算好"更有效。

记忆系统：跨越遗忘的鸿沟

LLM 有一个致命弱点：无状态（Stateless）。每次新会话对它来说都是白板一张。上次聊了什么、你喜欢什么风格、之前踩过什么坑——全部忘光。

Context Window 是硬瓶颈
即使模型号称支持 100 万 token，实际有效性在 ~25 万 token 后就开始衰减（Context Rot）。这意味着记忆系统不能简单地"把所有历史都塞进去"——必须选择性地检索和注入最相关的信息。

Planning：让 Agent 学会拆任务

简单任务（查天气、翻译一句话）不需要 Planning。但复杂任务（"帮我重构这个模块"、"写一篇研究报告"）需要 Agent 先想清楚要做什么、按什么顺序做。

Planning 的两种范式

Plan-then-Execute vs Interleaved Planning

OpenAI Codex 的执行计划

OpenAI 在 Harness Engineering 中使用的 Execution Plans（执行计划）是一个好例子：

// execution-plan.md — 检入代码仓库的计划文件

## 目标
实现用户认证模块

## 已完成
- [x] 数据库 Schema 设计
- [x] Repository 接口定义

## 当前
- [ ] JWT 工具函数 ← 进行中

## 待做
- [ ] Login Controller
- [ ] 单元测试
- [ ] 集成测试

## 决策日志
- 选择 Zod 做 schema 验证（模型自己选的）
- 放弃 passport.js，自己实现更可控

这个计划文件被版本控制，Agent 每次启动时读取，知道做到哪了、接下来干什么。这就解决了 LLM 无状态导致的"失忆重启"问题。

Planning 的核心价值
不是"让 Agent 有计划"，而是让计划成为可被检查、可被修改、可被持久化的实体。计划不在 Agent 的"脑子"里，而在文件系统里——任何人（人或 Agent）都能看到、修改、恢复。

多 Agent 协作

当任务足够复杂，一个 Agent 搞不定时，就需要多个 Agent 分工协作。这就像一个公司——CEO 不能既写代码又做设计还管运营。

三种协作模式

多 Agent 协作模式

何时需要多 Agent？

信号说明
任务需要不同"专业"编码 vs 设计 vs 写作，不同 system prompt 更合适
需要并行处理多个独立子任务可以同时跑
需要互相审查一个 Agent 写代码，另一个 Review——降低出错率
单 Agent 上下文不够任务太大，一个 Agent 的 context window 装不下
不要过早多 Agent
多 Agent 不是银弹。协调成本很高（通信延迟、信息丢失、任务边界模糊）。如果一个 Agent 就能搞定，不要拆成两个。OpenAI 的 Harness Engineering 实验中，大部分时间也是单 Agent 在工作，只在 Review 环节才引入第二个 Agent。

Harness Engineering：让 Agent 可靠

前面几章讲了 Agent 的能力：工具、记忆、规划、协作。但有能力 ≠ 可靠。Agent 在生产环境中最大的挑战不是"能不能做"，而是"做了会不会翻车"。

为什么"更强的模型"不够？

一个真实案例：某团队部署了一个无人监控的 Agent，凌晨三点它陷入无限重试循环，每分钟调用付费 API，到早上烧掉了 $50,000。

问题不在模型笨——模型很聪明。问题在于没有人设计"笼子"来兜住它犯错的情况。

Harness Engineering 三大支柱

核心理念：每次犯错都让系统变强

Mitchell Hashimoto（Terraform 创始人）的定义：

Harness Engineering 的本质
"每当你发现 Agent 犯了一个错误，你就花时间设计一个解决方案，使 Agent 永远不再犯同样的错误。"
日积月累，Harness 越来越健壮，Agent 能犯的错越来越少——这是一个正向复利飞轮。

从零手写一个 Agent

理论讲完了，现在动手。我们用最少的代码实现一个完整的 ReAct Agent——有工具、有循环、有验证。不用任何框架，只用原始 API。

最小 Agent 的骨架

import openai
import json

client = openai.OpenAI()

# ── 第1步：定义工具 ──────────────────────
tools = [
    {
        "type": "function",
        "function": {
            "name": "read_file",
            "description": "读取一个文件的内容",
            "parameters": {
                "type": "object",
                "properties": {
                    "path": {"type": "string"}
                },
                "required": ["path"]
            }
        }
    },
    {
        "type": "function",
        "function": {
            "name": "run_command",
            "description": "执行一条 shell 命令",
            "parameters": {
                "type": "object",
                "properties": {
                    "command": {"type": "string"}
                },
                "required": ["command"]
            }
        }
    }
]

# ── 第2步：工具执行函数 ──────────────────
def execute_tool(name, args):
    if name == "read_file":
        try:
            return open(args["path"]).read()
        except:
            return f"错误：文件不存在"

    elif name == "run_command":
        import subprocess
        result = subprocess.run(
            args["command"], shell=True,
            capture_output=True, text=True, timeout=30
        )
        return result.stdout + result.stderr

# ── 第3步：ReAct 主循环 ──────────────────
def run_agent(task, max_turns=10):
    messages = [
        {"role": "system", "content": """
你是一个编码助手。收到任务后：
1. 先思考需要做什么
2. 使用工具一步步完成
3. 每步都验证结果
4. 完成后说"任务完成"
"""},
        {"role": "user", "content": task}
    ]

    for turn in range(max_turns):
        # 调用 LLM
        response = client.chat.completions.create(
            model="gpt-4o",
            messages=messages,
            tools=tools
        )
        msg = response.choices[0].message

        # 如果 LLM 想调用工具
        if msg.tool_calls:
            messages.append(msg)  # 先把 assistant 消息加上

            for call in msg.tool_calls:
                # 执行工具
                result = execute_tool(
                    call.function.name,
                    json.loads(call.function.arguments)
                )
                # 把结果告诉 LLM
                messages.append({
                    "role": "tool",
                    "tool_call_id": call.id,
                    "content": result
                })

        # 如果 LLM 直接回复（不调用工具）→ 任务完成
        else:
            print(msg.content)
            return

    print("⚠️ 达到最大轮次")

# ── 运行！──────────────────────────────
run_agent("查看当前目录有哪些 Python 文件，然后统计总行数")

解剖这段代码

这 60 行代码就是一个完整的 Agent。它包含了所有核心组件：

代码 ↔ 概念映射

缺什么？

这个最小 Agent 能工作，但不适合生产环境。缺的就是之前讲的 Harness：

  • ❌ 没有记忆（关掉就忘光）
  • ❌ 没有错误恢复（工具出错就卡死）
  • ❌ 没有安全沙盒（run_command 能执行任何命令）
  • ❌ 没有上下文管理（对话太长就超 token 限制）
  • ❌ 没有验证循环（不检查输出质量）
  • ❌ 没有费用控制（可能无限循环烧钱）

补上这些，就是 Harness Engineering 的工作。

动手建议
先跑通这个最小版本，理解核心循环。然后一个一个加功能：先加错误处理，再加安全审批（command 执行前需要确认），再加记忆……每加一层你都会更深地理解 Agent 系统设计。

生产环境的挑战与实践

从"能跑"到"可靠地持续跑"之间，有一道鸿沟。这一章讲的是你在真实部署 Agent 时会遇到的问题。

六大生产级挑战

从 Demo 到 Production 的鸿沟

具体怎么应对

挑战问题解决方案
非确定性 同样的输入，LLM 每次输出不同 验证循环（Build → Test → Fix），用确定性测试兜底
幻觉 Agent 宣称"完成了"但其实没做 Pre-Completion Check：完成前强制跑测试
无限循环 Agent 反复尝试同一个失败的操作 Loop Detection + 最大步数限制 + 相同操作检测
Context Rot 对话太长导致模型"忘记"早期指令 上下文压缩（Compaction）+ 摘要 + 分段处理
安全风险 Agent 执行危险命令（rm -rf /） 沙盒隔离 + 命令审批 + 白名单
成本失控 无限调用 API 烧钱 Token 预算 + 超时 + 速率限制 + 告警

演进路线图

如果你要从零开始做一个生产级 Agent，建议这个顺序：

  1. Week 1 — 最小 ReAct 循环 + 2-3 个工具（上一章的代码）
  2. Week 2 — 加安全审批（命令执行前需要确认）+ 错误处理
  3. Week 3 — 加记忆系统（进度文件 + MEMORY.md）
  4. Week 4 — 加验证循环（Build → Test → Fix loop）
  5. Month 2 — 加可观测性（日志、指标、trace）+ 成本控制
  6. Month 3 — 多 Agent 协作 + 架构约束 Linter
最重要的一课
Agent 开发的 80% 工作量不在写 Agent 本身，而在写 Agent 周围的 Harness——验证、约束、恢复、监控。

这就是为什么 OpenAI 的工程师说："我们最大的挑战不再是写代码，而是设计让 AI 能可靠写代码的环境。"
