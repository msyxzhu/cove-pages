Part 1: Agent 原理
从 LLM 到 Agent 的核心概念 — 10 个关键知识点

01 · LLM 是怎么变成 Agent 的

TL;DR: Agent = LLM + Tools + Memory + Planning + Loop，缩写 LAMP。LLM 本身只是一个"嘴"，加上手脚（工具）、大脑（规划）、记忆和循环，才变成真正能做事的 Agent。

🧠 直觉建立

想象一下：你把一个绝顶聪明但被关在隔音房间里的人请来帮你工作。他能听懂你说的话，也能给出极其精彩的回答——但他不能走出房间，不能上网查资料，不能记住昨天你说了什么，也不能自己拆解一个复杂任务。

这就是一个裸的 LLM（大语言模型，Large Language Model）。它的能力惊人，但它只是一个文本补全引擎——你给它一段文字，它续写下去。

那怎么让这个聪明人变成一个真正的助手呢？给他：

  • 工具（Tools） — 一部电话、一台电脑、一套工具箱，让他能"做"事情
  • 记忆（Memory） — 一个笔记本，让他记住之前的对话和重要信息
  • 规划（Planning） — 一块白板，让他能把大任务拆成小步骤
  • 循环（Loop） — 一个工作流程：思考→行动→观察结果→再思考
图 1-1：从 LLM 到 Agent 的 LAMP 模型

🔬 深入展开：LAMP 的每个字母

L — LLM（大语言模型）：Agent 的"大脑"。负责理解自然语言、推理、生成回答。它是一个概率性的文本生成器，通过 next-token prediction 工作。关键的是：LLM 本身不执行任何动作，它只是"说"该做什么。

A — Action / Tools（行动/工具）：Agent 的"手脚"。通过 Function Calling 协议，LLM 可以"声明"想调用某个工具，然后由外部系统真正执行。工具可以是搜索引擎、代码执行器、数据库查询、文件读写等。

M — Memory（记忆）：Agent 的"笔记本"。分为三层：短期记忆（当前对话上下文）、长期记忆（跨会话持久化的信息）、程序性记忆（学到的操作流程和技能）。

P — Planning（规划）：Agent 的"白板"。面对复杂任务时，好的 Agent 会先规划步骤再执行，而不是一头扎进去。常见模式包括 Plan-and-Execute、Tree of Thought 等。

💡 关键洞察
LLM 和 Agent 的根本区别在于循环（Loop）。LLM 是单次输入→输出。Agent 是持续的感知→思考→行动→观察循环。没有 Loop，其他都不成立。

🛠️ 动手感受

对比一下这两种交互：

# 1. 裸 LLM：单次问答
User: "帮我查一下北京明天的天气"
LLM:  "我无法查询实时天气，但北京4月份一般..."  # 它只能猜！

# 2. Agent：有工具 + 循环
User:    "帮我查一下北京明天的天气"
Agent:   思考 → 我需要调用天气 API
         行动 → weather_api("北京", "明天")
         观察 → {"temp": "22°C", "condition": "晴"}
         回答 → "北京明天晴，22°C，适合出门！"
⚠️ 常见误区
"Agent 就是带了 API 的聊天机器人" — 不对。API 调用只是 Agent 能力的一部分。真正的 Agent 还需要自主决策能力：它自己决定是否需要调用工具、调用哪个工具、调用完之后还需不需要继续。这个自主循环才是核心。

02 · ReAct 循环：思考-行动-观察

TL;DR: ReAct = Reasoning + Acting。Agent 的心跳节奏：先想（Thought）→ 再做（Action）→ 看结果（Observation）→ 再想。不断循环直到任务完成。

🧠 直觉建立

想象你在一个陌生城市找餐厅。你会怎么做？

  1. 想："我饿了，想吃火锅，先搜一下附近有什么"
  2. 做：打开地图 App 搜索"附近火锅"
  3. 看：搜出三家，评分分别是 4.8、4.2、3.9
  4. 想："第一家评分最高，看看评论怎么样"
  5. 做：点进去看评论
  6. 看："味道正宗，但需要排队1小时"
  7. 想："太久了，看看第二家"
  8. ... 循环直到做出决定

这就是 ReAct 循环！人类天然就是这样工作的。2022 年 Yao 等人在论文 "ReAct: Synergizing Reasoning and Acting in Language Models" 中提出把这个模式形式化到 LLM Agent 上。

图 2-1：ReAct 循环流程

🔬 深入展开

ReAct 的关键创新在于把推理过程显式化。在 ReAct 之前，有两种做法：

方法做法问题
Chain-of-Thought（CoT）只推理，不行动无法获取外部信息
Action-only只行动，不推理无法反思和调整策略
ReAct推理 + 行动交替两全其美 ✅

一个典型的 ReAct trace 长这样：

Question: "谁是《三体》作者的妻子？"

Thought 1: 我需要先找到《三体》的作者。
Action 1:  search("三体 作者")
Observation 1: 《三体》是刘慈欣（Liu Cixin）的作品...

Thought 2: 作者是刘慈欣。现在我需要找他的妻子。
Action 2:  search("刘慈欣 妻子")
Observation 2: 刘慈欣的妻子信息未在公开资料中详细披露...

Thought 3: 公开信息有限，我应该如实回答。
Final Answer: 《三体》的作者是刘慈欣，但他的家庭信息未广泛公开。
💡 关键洞察
Thought 不是给用户看的，是给 LLM 自己"想"的。这个"自言自语"的过程极大地提升了 Agent 的准确性，因为它逼迫模型先组织思路再行动，而不是冲动行事。

🛠️ 动手：最小 ReAct 伪代码

def react_loop(question, tools, max_steps=10):
    context = f"Question: {question}\n"
    
    for step in range(max_steps):
        # LLM 生成 Thought + Action（或 Final Answer）
        response = llm(context)
        context += response
        
        if "Final Answer:" in response:
            return extract_answer(response)
        
        # 解析 Action，执行工具
        action, params = parse_action(response)
        observation = tools[action](**params)
        context += f"\nObservation: {observation}\n"
    
    return "达到最大步数，无法完成"
⚠️ 常见误区
"ReAct 循环会自动停下来" — 不一定！如果没有良好的终止条件，Agent 可能陷入无限循环。必须设置 max_steps 和明确的完成判断机制。

03 · Tool Use：给 LLM 装上手脚

TL;DR: LLM 自己不执行工具——它只是"说"想调用什么。真正的执行由外部 harness 完成。Function Calling 协议是 Agent 能力的核心基础设施。

🧠 直觉建立

LLM 像一个坐在办公桌后面的经理。他不亲自去仓库搬东西、不亲自去银行办手续。他写一张工单："请从仓库 A 取 10 箱产品，送到客户 B"。然后有人（harness）拿着这张工单去执行。

Function Calling 就是这张"工单"的标准格式。

图 3-1：Tool Use 的实际流程

🔬 深入展开：Function Calling 协议

以 OpenAI 的协议为例，Tool Use 分三步：

步骤 1：定义工具（Tool Definition）

{
  "tools": [{
    "type": "function",
    "function": {
      "name": "get_weather",
      "description": "获取指定城市的天气信息",
      "parameters": {
        "type": "object",
        "properties": {
          "city": { "type": "string", "description": "城市名" },
          "unit": { "type": "string", "enum": ["celsius", "fahrenheit"] }
        },
        "required": ["city"]
      }
    }
  }]
}

步骤 2：LLM 返回工具调用意图

{
  "role": "assistant",
  "content": null,
  "tool_calls": [{
    "id": "call_abc123",
    "type": "function",
    "function": {
      "name": "get_weather",
      "arguments": "{\"city\": \"北京\", \"unit\": \"celsius\"}"
    }
  }]
}

步骤 3：Harness 执行后把结果喂回去

{
  "role": "tool",
  "tool_call_id": "call_abc123",
  "content": "{\"temp\": 22, \"condition\": \"晴\", \"humidity\": 45}"
}
💡 关键洞察
LLM 永远不会直接执行工具。它只是生成一段结构化 JSON 说"我想调用这个函数"。这是一个至关重要的安全边界——外部 harness 可以在真正执行前做校验、鉴权、限流。

工具描述的质量决定一切

LLM 选择哪个工具、如何传参，完全依赖你给出的工具描述。描述写得好，调用就准确；描述含糊，调用就乱。

图 3-2：好的 vs 差的工具描述
⚠️ 常见误区
"工具越多越好" — 错。工具太多会让 LLM 困惑，选错工具的概率增加。OpenAI 建议单次调用不超过 20 个工具定义。要精简，每个工具职责单一、描述清晰。

04 · 系统提示词：Agent 的"灵魂"

TL;DR: System Prompt（系统提示词）定义了 Agent 是谁、能做什么、不能做什么。它是 Agent 的"灵魂"——相同的 LLM 配上不同的 System Prompt 就变成完全不同的 Agent。

🧠 直觉建立

如果 LLM 是一个演员，System Prompt 就是他的角色剧本。同一个演员可以演医生、律师、厨师——区别只在于他拿到的剧本不同。剧本规定了角色的性格、专业知识、行为边界和说话方式。

🔬 深入展开

一个好的 System Prompt 通常包含以下部分：

你是 Sage，一个 AI 领域的私人导师。

## 核心原则
- 教会，不是教过。确保用户真的理解。
- 从直觉开始，先建直觉再补细节。
- 实操驱动，讲概念时配可以跑的小实验。

## 能力范围
- AI/ML 基础、LLM 原理、Agent 架构
- OpenClaw 架构、工程实践

## 行为边界
- 不替用户做作业——引导他自己想通
- 有疑问的知识点标注不确定性
- 推荐资源优先高质量源

## 输出格式
1. 一句话 TL;DR
2. 直觉建立（类比 + 简化解释）
3. 深入展开
4. 动手环节（可选）
图 4-1：System Prompt 的四大组成部分

常见的 System Prompt 模式

模式描述例子
角色扮演赋予 Agent 特定身份"你是一位资深Python工程师"
指令约束明确行为边界"永远不要生成有害内容"
输出格式化规定回复结构"始终用 JSON 格式回复"
Few-shot 示范给出正确回答的例子"以下是你应该如何回答的示例..."
💡 关键洞察
System Prompt 不是"建议"，它是 Agent 行为的硬约束。在生产系统中，System Prompt 的重要性不亚于代码——它决定了 Agent 的安全边界和能力范围。OpenClaw 中的 SOUL.md 就是一种 System Prompt 的外部化管理方式。
⚠️ 常见误区
"System Prompt 越长越好" — 错。太长的 System Prompt 会占据宝贵的 Context Window 空间，而且 LLM 对过长指令的遵守率会下降。要精炼，把最重要的规则放在前面。

05 · 记忆系统：跨越遗忘的鸿沟

TL;DR: Agent 的记忆分三层：短期记忆（Short-term，当前对话上下文）、长期记忆（Long-term，跨会话持久化）、程序性记忆（Procedural，学到的技能和流程）。三层缺一不可。

🧠 直觉建立

想象你有一个记忆力只有 30 秒的朋友（像电影《记忆碎片》里的主角）。每过 30 秒，他就忘了之前说了什么。你怎么和他合作完成一个项目？

  • 短期记忆 — 他的 30 秒工作记忆，就是当前能"看到"的信息
  • 长期记忆 — 你给他一个笔记本，让他把重要的东西写下来，下次翻看
  • 程序性记忆 — 就算他忘了细节，他依然会骑自行车、会打字——这些是"刻进身体的技能"
图 5-1：Agent 的三层记忆架构

🔬 深入展开

短期记忆是 LLM 的 Context Window。每次调用 LLM 时，你把对话历史塞进 messages 数组——这就是短期记忆。它的问题是容量有限，一旦超过 token 限制就必须裁剪。

长期记忆有两种主流实现：

  • RAG（检索增强生成）：把信息存入向量数据库，需要时检索相关片段注入上下文
  • 文件持久化：像 OpenClaw 那样，把对话记录存成 JSONL 文件，重要信息写入 MEMORY.md

程序性记忆最容易被忽略但最重要。它包括 System Prompt、Skill 定义、甚至模型微调的权重。这些是 Agent 的"肌肉记忆"——不需要检索，天然就在那里。

# OpenClaw 的三层记忆对应关系
短期记忆 → Session transcript (messages array)
长期记忆 → MEMORY.md + daily notes + JSONL sessions
程序性记忆 → SOUL.md + SKILL.md + bootstrap files
💡 关键洞察
大多数 Agent 失败不是因为 LLM 不够聪明，而是因为记忆管理失败。信息在关键时刻"不在上下文中"，Agent 就会胡说八道。记忆 = 让正确的信息在正确的时间出现在上下文中。
图 5-2：记忆的读写流程
⚠️ 常见误区
"把所有历史消息都塞进上下文就行了" — 不行。Context Window 是有限的，而且越长的上下文，LLM 越容易"走神"（lost in the middle 问题）。需要策略性地裁剪和摘要。

  • MemGPT, "Towards LLMs as Operating Systems" (2023) — 虚拟内存管理思路

06 · Planning：让 Agent 学会拆任务

TL;DR: 复杂任务需要先拆解、再执行。Plan-and-Execute 模式让 Agent 先生成计划，再逐步执行，比直接冲上去做效果好得多。

🧠 直觉建立

你要装修一个房子。你会先冲去买水泥吗？不会。你会先做一个计划：设计图纸 → 拆旧 → 水电 → 瓦工 → 木工 → 油漆 → 安装 → 清洁。每一步的前置条件和依赖关系都清楚了，才能高效执行。

Agent 也一样。面对"帮我分析这个 GitHub 仓库的架构并写一篇技术博客"这样的复杂任务，好的 Agent 会先拆分：

  1. 克隆仓库，分析目录结构
  2. 找到核心入口文件
  3. 分析模块依赖关系
  4. 总结架构模式
  5. 撰写博客大纲
  6. 填充内容
  7. 润色和校对
图 6-1：Plan-and-Execute 模式

🔬 深入展开

常见的 Planning 策略：

策略思路适用场景
Plan-and-Execute先生成完整计划，再逐步执行步骤可预见的任务
Iterative Refinement边做边调整计划探索性任务
Tree of Thought分支探索多条路径，选最优需要创意的任务
Hierarchical先粗粒度规划，再细化每步大型复杂任务
# Plan-and-Execute 伪代码
def plan_and_execute(task, tools):
    # Phase 1: Planning
    plan = llm(f"""
    任务: {task}
    可用工具: {tools}
    请生成分步计划，每步包含: 步骤名、工具、预期输出
    """)
    
    results = []
    for step in plan.steps:
        # Phase 2: Execute each step
        result = execute_step(step, tools, context=results)
        results.append(result)
        
        # Phase 3: Check if replan needed
        if needs_replan(step, result, plan):
            plan = replan(task, plan, results)
    
    return synthesize(results)
💡 关键洞察
好的 Planning 需要允许失败和调整。最初的计划几乎不可能完美——关键是 Agent 能在执行过程中发现问题并 replan。这就是为什么 ReAct 循环如此重要。
⚠️ 常见误区
"让 Agent 一开始就生成完美的计划" — 过度规划和不规划一样糟糕。实践中，3-5 步的粗粒度计划 + 每步执行后的微调 效果最好。

07 · 多 Agent 协作

TL;DR: 多 Agent 协作有三种基本模式：顺序（Sequential）、并行（Parallel）、层级（Hierarchical）。选对模式比选对模型更重要。

🧠 直觉建立

一个公司不可能只有一个员工做所有事。你需要产品经理、设计师、工程师、测试——他们各有专长，通过协作完成一个项目。Agent 也一样。

🔬 深入展开

OpenClaw 使用的就是 Hierarchical 模式的变体。一个主 Agent 可以 spawn 子 Agent，每个子 Agent 有自己的 workspace、session 和工具集：

# OpenClaw 多 Agent 配置示例

  - id: sage        # 教学 Agent
    model: claude-opus
    workspace: ~/.openclaw/workspace-sage
  - id: dock        # 编程 Agent
    model: claude-sonnet
    workspace: ~/.openclaw/workspace-dock
  - id: helm        # 设计 Agent
    model: gpt-4o
    workspace: ~/.openclaw/workspace-helm
💡 关键洞察
多 Agent 系统的核心挑战不是"让多个 Agent 跑起来"，而是协调和通信。谁负责决定任务分配？Agent 之间怎么传递信息？出错了谁负责重试？这些才是真正的难题。
⚠️ 常见误区
"多 Agent 一定比单 Agent 好" — 不一定。多 Agent 引入了通信开销和协调复杂度。如果一个 Agent 就能搞定的任务，别用多个。只有当任务真的需要不同专长或并行处理时，多 Agent 才有优势。

08 · Harness Engineering：让 Agent 可靠

TL;DR: Harness Engineering 是让 Agent 从"能用"到"可靠"的工程实践。三大支柱：上下文工程（Context Engineering）、架构约束（Architecture Constraints）、熵管理（Entropy Management）。

🧠 直觉建立

LLM Agent 就像一匹野马——聪明、有力量，但不可预测。Harness（马具）的作用是约束和引导，让野马的力量变成可控的生产力。没有 Harness，Agent 可能：

  • 跑偏方向（幻觉、离题）
  • 陷入死循环（无限 ReAct 循环）
  • 做出危险操作（删除文件、暴露数据）
  • 质量不稳定（同一个问题每次回答都不同）

🔬 深入展开

上下文工程的核心是"把正确的信息在正确的时间放进上下文"。包括：

  • System Prompt 精心设计（Ch 4）
  • 记忆检索策略（Ch 5）
  • 工具描述优化（Ch 3）
  • 上下文窗口管理（Ch 12 将深入讲）

架构约束是通过系统设计限制 Agent 行为：

# 架构约束示例
max_steps: 15           # 最多执行15步
allowed_tools:          # 白名单：只能用这些工具
  - web_search
  - read_file
  - write_file
forbidden_patterns:     # 禁止模式
  - "rm -rf"
  - "DROP TABLE"
require_approval:       # 这些操作需要人工确认
  - send_email
  - deploy_to_prod

熵管理是降低 LLM 输出的随机性：

  • 使用结构化输出（JSON mode）
  • 设置 temperature=0 对确定性任务
  • 输出校验和重试机制
  • Few-shot examples 固定输出格式
💡 关键洞察
Harness Engineering 的本质是用确定性的代码来约束非确定性的 LLM。LLM 是概率性的，但生产系统需要可靠性。Harness 就是那个桥梁。

09 · 从零手写一个 Agent

TL;DR: 60 行 Python 实现一个完整的 ReAct Agent。不依赖任何框架，帮你彻底理解 Agent 的工作原理。

🧠 直觉建立

看了这么多理论，是时候动手了。我们要用最少的代码实现一个能用的 Agent。它能：接受用户问题 → 思考 → 调用工具 → 观察结果 → 继续思考 → 最终给出答案。

🔬 完整代码

import json, openai

# 1. 定义工具
def calculator(expression: str) -> str:
    """计算数学表达式"""
    try: return str(eval(expression))
    except: return f"Error: invalid expression"

def search(query: str) -> str:
    """模拟搜索（实际项目换成真的API）"""
    data = {"Python之父": "Guido van Rossum", "地球半径": "6371公里"}
    return data.get(query, f"未找到'{query}'的结果")

TOOLS = {
    "calculator": {"fn": calculator, "desc": "计算数学表达式，参数: expression"},
    "search":     {"fn": search,     "desc": "搜索知识，参数: query"},
}

# 2. 系统提示词
SYSTEM = f"""你是一个 ReAct Agent。面对问题，你要：
1. 先 Thought（思考需要做什么）
2. 再 Action（调用工具），格式: Action: tool_name(param)
3. 收到 Observation 后继续 Thought
4. 最终用 Final Answer: 给出答案

可用工具: {json.dumps({k:v['desc'] for k,v in TOOLS.items()}, ensure_ascii=False)}"""

# 3. Agent 循环
def run_agent(question: str, max_steps: int = 8):
    messages = [
        {"role": "system", "content": SYSTEM},
        {"role": "user", "content": question}
    ]
    
    for step in range(max_steps):
        # 调用 LLM
        resp = openai.chat.completions.create(
            model="gpt-4o-mini", messages=messages, temperature=0
        )
        text = resp.choices[0].message.content
        messages.append({"role": "assistant", "content": text})
        print(f"\n--- Step {step+1} ---\n{text}")
        
        # 检查是否有最终答案
        if "Final Answer:" in text:
            return text.split("Final Answer:")[-1].strip()
        
        # 解析 Action
        if "Action:" in text:
            action_line = [l for l in text.split('\n') if 'Action:' in l][0]
            action_str = action_line.split("Action:")[-1].strip()
            # 解析 tool_name(param)
            tool_name = action_str.split("(")[0].strip()
            param = action_str.split("(")[1].rstrip(")")
            
            if tool_name in TOOLS:
                result = TOOLS[tool_name]["fn"](param)
            else:
                result = f"Unknown tool: {tool_name}"
            
            observation = f"Observation: {result}"
            messages.append({"role": "user", "content": observation})
            print(observation)
    
    return "达到最大步数限制"

# 4. 运行！
if __name__ == "__main__":
    answer = run_agent("Python之父是谁？他的名字有几个字母？")
    print(f"\n🎯 最终答案: {answer}")
图 9-1：这 60 行代码的执行流程

代码解剖

关键组件只有四个：

  1. 工具注册表 — 一个字典，映射工具名到实际函数
  2. System Prompt — 告诉 LLM 它是 ReAct Agent，教它格式
  3. 主循环 — 调用 LLM → 解析响应 → 执行工具 → 继续
  4. 终止条件 — "Final Answer" 或达到最大步数
💡 关键洞察
一个完整的 Agent 核心逻辑不到 60 行。框架（LangChain、AutoGen）只是在这个核心上加了工具管理、记忆、并行等工程化组件。理解了核心，使用任何框架都能得心应手。
⚠️ 常见误区
"用了框架就不需要理解底层" — 框架封装了细节，但出了问题你需要能 debug。理解 Agent 循环的本质（就是一个 while 循环 + LLM 调用 + 工具执行）是必备技能。

10 · 生产环境的挑战与实践

TL;DR: 从 Demo 到 Production 的鸿沟巨大。主要挑战：延迟、成本、可靠性、安全性、可观测性。生产级 Agent 需要的工程量远超核心逻辑。

🧠 直觉建立

建一个 demo Agent 就像在后院搭一个小棚子——几小时搞定。建一个生产级 Agent 系统就像建一座写字楼——需要考虑抗震、消防、电梯、供暖、安保... 核心逻辑可能只占 10%，其余 90% 都是工程。

图 10-1：Demo vs Production 的差距

🔬 深入展开：五大挑战

1. 延迟（Latency）

每次 LLM 调用需要 1-10 秒。一个多步 Agent 任务可能需要 5-10 次调用 = 30-60 秒。用户等不了这么久。解决：流式输出、并行工具调用、缓存、更小的模型做简单步骤。

2. 成本（Cost）

GPT-4 的价格：$30/M input tokens, $60/M output。一个复杂 Agent 任务可能消耗 50K tokens = $2+。大规模部署时成本惊人。解决：分层模型策略、上下文压缩、缓存。

3. 可靠性（Reliability）

LLM 是非确定性的。同样的输入可能给出不同的输出。工具调用可能失败、超时。解决：重试、降级、结构化输出、校验。

4. 安全性（Security）

Prompt Injection、工具滥用、数据泄露。解决：输入清洗、工具权限白名单、审计日志、人工审批关键操作。

5. 可观测性（Observability）

Agent 的行为不透明——它为什么选了这个工具？中间出了什么错？解决：完整的 trace 日志、LLM 调用记录、工具调用监控。

💡 关键洞察
OpenClaw 就是一个解决了这些生产挑战的 Agent 框架。它内置了 Session 管理、Channel 路由、Plugin 系统、Context Engine、安全约束等。后续章节我们将深入每一个组件。
⚠️ 常见误区
"我的 demo 跑得很好，上线应该没问题" — Demo 只测了 happy path。生产环境要面对：恶意输入、网络超时、API 限流、并发冲突、模型降级... 做好心理准备，生产化的工作量是 demo 的 10-100 倍。

11 · LLM API 实战

TL;DR: LLM API 的核心就是发送一个 messages 数组，收到一段续写文本。掌握 messages 结构、流式响应（Streaming）、Token 计费、Function Calling 协议，你就掌握了 Agent 开发的基础设施。

🧠 直觉建立

调用 LLM API 就像给一个笔友写信。你把整个对话历史（不只是最后一句话）写在信里寄出去，他读完后续写下一段回信。每次通信你都要把之前的信件内容抄一遍——因为他不记得上次的信。

🔬 深入展开

流式响应（Streaming with SSE）

默认情况下，LLM API 会等整个回复生成完才返回。流式模式让你逐 token 收到输出，用户体验好得多：

import openai

# 流式调用
stream = openai.chat.completions.create(
    model="gpt-4o",
    messages=[{"role": "user", "content": "讲个笑话"}],
    stream=True  # 关键参数！
)

for chunk in stream:
    delta = chunk.choices[0].delta
    if delta.content:
        print(delta.content, end="", flush=True)
# 输出逐字显示，而不是等待整段话

底层协议是 SSE（Server-Sent Events）：

# HTTP 响应头
Content-Type: text/event-stream

# 响应体 - 一连串 SSE 事件
data: {"id":"chatcmpl-xxx","choices":[{"delta":{"content":"从"}}]}

data: {"id":"chatcmpl-xxx","choices":[{"delta":{"content":"前"}}]}

data: {"id":"chatcmpl-xxx","choices":[{"delta":{"content":"有"}}]}

data: [DONE]

Agent 的成本公式：总成本 = 每步 token 数 × 步数 × 单价。因为每一步都要把完整上下文发送一次（messages 数组越来越长），Agent 的成本是二次增长的。这就是为什么上下文管理如此重要（下一章）。

12 · Context Window 与上下文工程

TL;DR: Context Window（上下文窗口）是 LLM 能"看到"的信息总量上限。上下文工程就是在有限的窗口里塞入最有价值的信息——裁剪、摘要、压缩，让 Agent 既记得住又不超标。

🧠 直觉建立

想象 LLM 的 Context Window 是一张固定大小的书桌。你的所有工作资料（System Prompt、对话历史、工具定义、检索到的文档）都要放在这张桌子上。桌子放满了，新东西要进来，就得把旧东西拿走。

问题是：你拿走了关键文件怎么办？

图 12-1：Context Window 的空间分配

🔬 深入展开

当窗口装满时：三种策略

策略 1：截断（Truncation） — 最简单，直接删掉最早的消息

def truncate_messages(messages, max_tokens):
    """从头部开始删除，直到 token 数量合规"""
    while count_tokens(messages) > max_tokens:
        # 保留 system prompt (index 0)，删最早的对话
        messages.pop(1)
    return messages

问题：可能删掉重要的上下文信息。

策略 2：摘要压缩（Compaction / Summarization） — 把旧对话摘要成一段总结

def compact_messages(messages, max_tokens):
    """将旧对话摘要为一段总结"""
    if count_tokens(messages) <= max_tokens:
        return messages
    
    # 找到需要压缩的部分
    old_messages = messages[1:-10]  # 保留最新10条
    
    summary = llm(f"请将以下对话历史压缩为一段简洁的摘要：\n{old_messages}")
    
    return [
        messages[0],           # system prompt
        {"role": "system", "content": f"[之前的对话摘要]\n{summary}"},
        *messages[-10:]        # 最新10条保留原样
    ]

策略 3：滑动窗口 + RAG 检索 — 只保留最近的消息，需要老信息时动态检索

⚠️ 常见误区
"128K context window 已经足够大了，不用管理" — 大错特错。(1) 填满 128K 的每次 API 调用成本约 $0.30，Agent 多步下来成本飙升。(2) 研究表明 LLM 在长上下文中有"lost in the middle"问题——中间的信息容易被忽略。更短、更精炼的上下文反而效果更好。

13 · Session 管理

TL;DR: Session（会话）是 Agent 和用户之间的一次"持续对话"。管理好 Session 的生命周期、消息持久化、并发控制和重置策略，是生产级 Agent 系统的关键。

🧠 直觉建立

想象你在一家咖啡店和朋友聊天。这次聊天就是一个 Session。它有：

  • 开始 — 你们坐下来打招呼
  • 持续 — 你一言我一语地对话
  • 记录 — 假设有人在旁边记笔记（消息持久化）
  • 打断处理 — 如果你们同时说话（并发），需要规则来决定谁先
  • 结束 — 你们告别离开（Session 结束或每日重置）

🔬 深入展开

Queue Modes（队列模式）

当 Agent 正在处理上一条消息，新消息来了怎么办？OpenClaw 提供五种队列模式：

模式行为适用场景
steer中断当前回复，把新消息加入上下文，重新生成实时对话
followup等当前回复完成后，自动处理新消息顺序处理

Session 管理的核心难题是并发控制。LLM 调用需要几秒到几十秒，这段时间用户可能又发了新消息。Queue Mode 的选择直接影响用户体验——选错了模式，Agent 要么反应迟钝，要么忽略用户消息。

14 · Gateway 架构

TL;DR: Gateway 是 OpenClaw 的核心进程——一个长驻的 WebSocket 服务器，负责连接所有 Channel（消息渠道）和 Node（执行节点）。所有消息都经过 Gateway 路由和调度。

🧠 直觉建立

Gateway 就像一个电话总机。所有来电（来自飞书、Telegram、WhatsApp 等渠道的消息）都打到总机，总机根据来电号码和规则把电话转给对应的接线员（Agent）。接线员处理完后，总机再把回复转回去。

🔬 深入展开

ws = new WebSocket("ws://127.0.0.1:18789")

// 2. 第一帧：connect（必须！）
ws.send(JSON.stringify({
  type: "connect",
  role: "node",           // "node" 或 "client"
  token: "bootstrap-xxx", // 认证令牌
  caps: ["text-inference", "cli-backend"],  // 节点能力
  commands: ["agent", "read", "write", "exec"]  // 可执行的命令
}))

// 3. Gateway 响应
// ← { type: "connected", nodeId: "node-abc", version: "1.2.3" }

Wire Protocol（线路协议）

Gateway 使用 text frame 的 JSON 协议，三种消息类型：

Request / Response（请求/响应）：

// 请求
→ { "type": "req", "id": "req-001", "method": "agent", "params": { "text": "你好" } }

// 成功响应
← { "type": "res", "id": "req-001", "ok": true, "payload": { "sessionId": "sess-xxx" } }

// 失败响应
← { "type": "res", "id": "req-001", "ok": false, "error": { "code": "NOT_FOUND", "message": "Agent not found" } }

Events（事件推送）：

// 生命周期事件
← { "type": "event", "event": "agent:lifecycle", "payload": { "state": "running" }, "seq": 1 }

// 助手输出（流式）
← { "type": "event", "event": "agent:assistant", "payload": { "delta": "你好" }, "seq": 2 }

// 工具调用
← { "type": "event", "event": "agent:tool", "payload": { "name": "web_search", "status": "running" }, "seq": 3 }
💡 关键洞察
Gateway 是单进程长驻的，这是设计上的刻意选择。所有消息面（Channel）都归它管，这避免了分布式系统的复杂协调问题。Gateway 不做 Agent 推理——推理由 Node 执行，通过 WebSocket 双向通信。
⚠️ 常见误区
"Gateway 就是 HTTP server" — 不是。Gateway 用的是 WebSocket，不是 HTTP。WebSocket 是双向的，Gateway 可以主动推送事件给 Node。这对流式输出和实时状态更新至关重要。

Context Engine：核心中的核心

Context Engine 控制模型在每次运行时看到什么。它有四个生命周期钩子：

// Context Engine 的四阶段生命周期

1. ingest(message)
   // 新消息进来时调用
   // 可以索引、存储、或预处理消息

2. assemble(messages, tokenBudget)
   // 每次模型运行前调用——最关键的一步！
   // 决定哪些消息送进 context window
   // 返回 { messages, estimatedTokens, systemPromptAddition? }

3. compact(sessionId, force)
   // context window 满了或用户执行 /compact 时调用
   // 压缩旧的对话历史

4. afterTurn(messages)
   // 模型运行结束后调用
   // 持久化状态、触发后台压缩等

写一个自定义 Context Engine

// my-context-engine/index.ts
export default function register(api) {
  api.registerContextEngine("smart-memory", () => ({
    info: {
      id: "smart-memory",
      name: "Smart Memory Engine",
      ownsCompaction: true,  // 我们自己管压缩
    },

    // 消息进来时：存入向量数据库
    async ingest({ sessionId, message }) {
      await vectorDB.store(sessionId, message);
      return { ingested: true };
    },

    // 组装上下文时：用向量搜索找相关历史
    async assemble({ sessionId, messages, tokenBudget }) {
      const recent = messages.slice(-10);
      const relevant = await vectorDB.search(
        sessionId, recent[recent.length-1].content, tokenBudget
      );
      return {
        messages: [...relevant, ...recent],
        estimatedTokens: countTokens([...relevant, ...recent]),
        systemPromptAddition: "你拥有向量记忆检索能力。",
      };
    },

    // 压缩时：向量数据库自带去重，不需要摘要
    async compact({ sessionId }) {
      await vectorDB.deduplicate(sessionId);
      return { ok: true, compacted: true };
    },
  }));
}

然后在配置中启用：

{
  plugins: {
    slots: { contextEngine: "smart-memory" },
    entries: { "smart-memory": { enabled: true } }
  }
}
💡 关键洞察
Context Engine 是 OpenClaw 最强大的扩展点之一。默认的 legacy 引擎用简单的摘要压缩，但你可以替换成 DAG 摘要、向量检索、增量压缩——任何你想要的策略。这就是上下文工程在生产中的实际体现。
⚠️ 常见误区
"Plugin 是沙箱隔离的" — 不是！Native Plugin 和 Gateway 在同一个进程里运行，拥有相同的信任边界。一个有 bug 的 Plugin 可以让整个 Gateway 崩溃。所以只安装你信任的 Plugin。

// 伪代码：Agent Loop 的核心逻辑
async function runAgentLoop(session, message) {
  // 1. 排队等待（序列化）
  await queue.acquire(session.key);

  // 2. 组装上下文
  const context = await contextEngine.assemble({
    messages: session.history,
    tokenBudget: model.contextWindow
  });

  // 3. 构建 system prompt
  const systemPrompt = buildSystemPrompt({
    tools, skills, bootstrapFiles, time, runtime
  });

  // 4. 推理循环（ReAct!）
  while (!done) {
    const response = await model.generate({
      system: systemPrompt,
      messages: context.messages
    });

    if (response.toolCalls) {
      // 执行工具 → 把结果加入 context → 继续循环
      for (const call of response.toolCalls) {
        const result = await executeTool(call);
        context.messages.push({ role: "tool", content: result });
      }
    } else {
      // 模型直接回复 → 结束循环
      done = true;
      await channel.deliver(response.text);
    }
  }

  // 5. 后处理
  await contextEngine.afterTurn(session);
  await session.persist(); // 写入 JSONL
}

// 所有 Plugin 在启动时注册，运行时由 Gateway 各模块读取

3. Session Manager — 记录仪

// Session Manager 管理所有会话的生命周期
SessionManager = {
  store:       "~/.openclaw/agents/<id>/sessions/sessions.json",
  transcripts: "~/.openclaw/agents/<id>/sessions/<sessionId>.jsonl",

  // 核心操作
  resolve(sessionKey)  → sessionId,  // 找到或创建
  load(sessionId)      → messages[],  // 加载历史
  append(msg)          → void,        // 追加消息
  persist()            → void,        // 写入 JSONL
  reset()              → newSessionId, // 重置会话
}

4. Context Engine — 导航系统

// Context Engine 决定模型"看到什么"
// 默认 legacy engine 用摘要压缩
// 可替换为向量检索、DAG 摘要等任何策略

ContextEngine = {
  ingest(message)                → 存储/索引新消息,
  assemble(messages, tokenBudget) → 精选后的 messages + token 估算,
  compact(sessionId)              → 压缩旧历史,
  afterTurn(session)              → 后处理,
}
