11 · LLM API 实战

TL;DR:LLM 的核心 API 就一个--发送 messages 数组,接收 completion。掌握 streaming、function calling、token 计费就够了。

Messages 数组结构

// 这是 LLM API 最核心的数据结构
{
  "model": "gpt-4",
  "messages": [
    { "role": "system",    "content": "你是一个助手" },   // 系统指令
    { "role": "user",      "content": "你好" },          // 用户消息
    { "role": "assistant", "content": "你好!有什么帮助?" }, // AI 历史回复
    { "role": "user",      "content": "今天天气?" }       // 当前问题
  ],
  "temperature": 0.7,     // 随机性(0=确定性,1=创造性)
  "max_tokens": 1000,      // 最大输出长度
  "stream": true           // 是否流式输出
}

Token 与计费

// Token ≈ 3/4 个英文单词 ≈ 1-2 个中文字

"Hello world"          → 2 tokens
"人工智能"              → 2-3 tokens
一页 A4 文字 (中文)     ≈ 400-600 tokens

// 计费 = 输入 token × 输入价 + 输出 token × 输出价
// GPT-4 大约:输入 $30/M tokens, 输出 $60/M tokens
// Claude Opus:输入 $15/M, 输出 $75/M

12 · Attention 机制 + Context Window

TL;DR:Attention 的基础原理在 0C 章讲过了。这一章聚焦工程实战:KV Cache 为什么吃显存、Context Window 满了怎么办、以及为什么重要信息不能放在中间。

📌 回顾
我们在 Chapter 0C 已经从直觉层面理解了 Attention(Q/K/V 图书馆类比)和 Transformer 架构。如果你对 Q·K^T/√d → softmax → ×V 这个流程还不熟悉,建议先回顾 0C。

这一章从工程和面试角度深入--KV Cache 优化、Context Window 管理策略、以及 Lost in the Middle 现象。

KV Cache(键值缓存)--为什么长对话吃显存?

先看一个问题:当模型生成第 100 个 token 时,它需要对前 99 个 token 都算一遍 Attention。生成第 101 个 token 时,又要对前 100 个 token 算一遍。这里有大量重复计算--前 99 个 token 的 K 和 V 向量,在生成第 100 和第 101 个 token 时完全一样,为什么要算两次?

// 没有 KV Cache:每生成一个新 token,重新计算所有 K/V

生成 token 1:  计算 K1, V1
生成 token 2:  计算 K1, V1, K2, V2           ← K1V1 重复算了!
生成 token 3:  计算 K1, V1, K2, V2, K3, V3   ← K1V1, K2V2 重复算了!
...
生成 token N:  计算 K1..Kn, V1..Vn            ← O(N2) 的重复计算

// 有 KV Cache:只算新 token 的 K/V,旧的从缓存读

生成 token 1:  计算 K1, V1 → 存入缓存
生成 token 2:  读缓存 K1, V1 + 计算 K2, V2 → 存入缓存
生成 token 3:  读缓存 K1..2, V1..2 + 计算 K3, V3 → 存入缓存
...
生成 token N:  读缓存 K1..n-1, V1..n-1 + 计算 Kn, Vn  ← O(N) 新计算

类比:想象一个法庭书记员。没有 KV Cache 就像每次有人发言,书记员都要从头把整个庭审记录重新读一遍再记新内容。有了 KV Cache,书记员保持着自己的笔记,每次只需要在笔记末尾追加新内容。

⚠️ KV Cache 的代价:显存
KV Cache 用空间换时间。每个 token 需要缓存所有层、所有 head 的 K 和 V 向量。

粗算:一个 7B 模型,32 层 × 32 heads × 128 维 × 2(K+V)× 2 字节(FP16)= 每个 token 约 0.5 MB
一个 128K context window 的对话 → KV Cache 约 64 GB 显存!

这就是为什么你和 AI 聊天聊久了会看到"上下文已满"--很多时候不是模型"忘了",而是 KV Cache 的显存用完了。

Context Window 管理策略

Context Window(上下文窗口)是模型一次能处理的最大 token 数。GPT-4o 是 128K,Claude 是 200K。听起来很大,但实际上要精打细算:

// Context Window 的预算分配(Budget Allocation)

┌──────────────────────────────────────────────────┐
│                 128K Context Window               │
├────────────┬───────────┬──────────────┬───────────┤
│ System     │ Tools /   │ Conversation │ Current   │
│ Prompt     │ Skills    │ History      │ Query     │
│ ~20%       │ ~15%      │ ~50%         │ ~15%      │
│ (25K)      │ (19K)     │ (64K)        │ (19K)     │
├────────────┴───────────┴──────────────┴───────────┤
│ System Prompt: 角色定义、规则、SOUL.md 等          │
│ Tools/Skills: 工具定义、技能文件内容              │
│ History: 之前的对话消息                          │
│ Current: 用户当前的问题 + 本轮生成的回答          │
└──────────────────────────────────────────────────┘

当 Context Window 快满了,怎么办?四种策略:

策略做法优点缺点适用场景
截断
(Truncation)		 丢弃最旧的消息 实现简单,零延迟 可能丢失关键上下文 闲聊、低风险对话
摘要
(Summarization)		 用 LLM 把旧对话压缩成摘要 保留关键信息,压缩率高 摘要可能遗漏细节,有额外 API 调用成本 长期项目、多轮任务
滑动窗口
(Sliding Window)		 只保留最近 N 条消息 行为可预测,实现简单 N 之前的信息全部丢失 客服对话、短任务
选择性保留
(Selective Retention)		 保留重要消息(工具结果、关键决策),丢弃闲聊 在有限空间内最大化信息密度 需要判断"什么是重要的" Agent 任务、编程辅助 
// OpenClaw 的 Context Engine 工作原理(简化版)

function assemble(msgs, budget) {
  const system = msgs.filter(m => m.role === 'system')  // 永远保留
  const recent = msgs.slice(-10)                        // 最近 10 条必须保留
  let remaining = budget - tokenCount(system) - tokenCount(recent)

  // 中间的消息按重要性排序
  const middle = msgs.slice(system.length, -10)
    .map(m => ({ ...m, score: importance(m) }))
    .sort((a, b) => b.score - a.score)

  // 贪心装入:优先保留重要的消息
  const selected = []
  for (const m of middle) {
    if (tokenCount(m) <= remaining) {
      selected.push(m)
      remaining -= tokenCount(m)
    }
  }

  // 按原始顺序重新排列,保持对话连贯性
  return [...system, ...selected.sort(byIndex), ...recent]
}

// importance() 的评分规则:
//   tool_result(工具调用结果)→ 高分(包含事实数据)
//   user 消息带代码/文件 → 高分(可能是关键上下文)
//   assistant 的短回答 → 低分("好的"、"明白了")
//   user 的闲聊 → 低分("谢谢"、"嗯")

Lost in the Middle(中间丢失效应)

2023 年,斯坦福的 Liu et al. 发表了一篇重要论文 "Lost in the Middle: How Language Models Use Long Contexts",揭示了一个关键现象:模型对上下文中不同位置的信息,关注度差异巨大。

// Lost in the Middle:注意力权重在不同位置的分布

位置:  [开头] ────────── [中间] ────────── [结尾]
权重:  ████              ██                ████████

开头(system prompt)→ 权重高(位置编码优势)
中间(旧的对话历史)→ 权重低(最容易被"遗忘")
结尾(最近的消息)  → 权重高(近因效应)

// 这就是为什么多轮对话越长,Agent 越容易"忘记"中间说过的话
💡 工程实践建议
1. 重要信息放两头:System Prompt(开头)放角色定义和核心规则,最近的消息(结尾)放当前任务上下文。不要把关键指令埋在长对话的中间。

2. RAG 检索结果要排序:如果你检索了 5 篇相关文档,把最相关的放在第一个,而不是随机排列。模型会更关注第一篇和最后一篇,中间的容易被忽略。这就是为什么 RAG 需要 Reranker(重排序器)。

3. Compaction（压缩）优于 Truncation（截断）:与其直接丢弃中间的消息,不如把它们压缩成摘要放在开头位置。OpenClaw 的做法是:当 context 使用超过 80%,自动触发 compaction,把旧对话压缩为一段总结。

分层压缩机制——生产级的 Context 管理

上面的 4 种策略是基础，但生产级 Agent 需要更精细的分层压缩。核心思想：不是一刺切地压，而是按层次、按价值压。

  分层压缩流程（从微观到宏观）：

  Layer 1: Micro 压缩（单条消息级别）
  │  • 工具返回的大段 JSON/HTML → 只保留关键字段
  │  • 已完成的工具调用结果 → 压缩为一行摘要
  │  • 每个工具输出设硬上限（如 2000 token）
  │
  ▼
  Layer 2: Auto 压缩（对话片段级别）
  │  • 当 context 使用超过阈值（如 80%）时触发
  │  • 用小模型（如 GPT-4o-mini）对旧对话生成摘要
  │  • 摘要放回 context 开头位置（利用 U 形注意力）
  │
  ▼
  Layer 3: 重注入（压缩后必须做！）
     • 压完后重新注入 System Prompt 的核心身份
     • 重新注入当前任务目标（避免压完就忘了自己在干嘛）
     • 保留最近 N 条消息不压缩（短期工作记忆）
💡 为什么压完要“重注入”？
这是新手最容易忽略的坑。压缩会丢失信息——如果你的 System Prompt 里说“你是一个客服助手，只能回答产品相关问题”，但压缩时把这段抹掉了，Agent 可能突然“失忆”开始回答无关问题。

OpenClaw 的做法：压缩后自动重新注入 SOUL.md（角色定义）和当前任务上下文，确保 Agent “压完还记得自己是谁、在干什么”。
🎯 面试高频题
Q: Attention 本质是什么?从向量空间变换角度解释。

标准答案:Attention 本质是一个加权信息聚合机制。每个 token 通过 Q/K/V 三个线性变换映射到不同的向量空间:Q(查询)代表"我在找什么",K(键)代表"我有什么",V(值)代表"我的内容"。通过 Q·K^T 计算相关度,softmax 归一化后加权求和 V,得到融合了全局信息的表示。数学上:Attention(Q,K,V) = softmax(QK^T/√d_k)V。

加分回答:提到 Multi-Head Attention(多个 head 关注不同方面,如语法/语义/指代),√d_k 缩放防止梯度消失,以及 KV Cache 如何优化推理速度。

常见踩坑:❌ 只说"Attention 让模型关注重要的部分"--太模糊,要说出 Q/K/V 的计算过程和 KV Cache 的工程意义。

🎯 面试高频题
Q: 多轮对话的存储、记忆解决方案?

标准答案:短期记忆存在 context window 里(messages 数组),长期记忆持久化到文件/数据库。具体实现:1JSONL 文件存储完整对话历史2context window 满时用 compaction 压缩旧消息为摘要3重要信息主动写入 Memory 文件。OpenClaw 的做法:session transcript 用 JSONL,长期记忆用 MEMORY.md + memory/YYYY-MM-DD.md。

加分回答:提到 dmScope 配置(main/per-peer/per-channel-peer)来隔离不同用户的会话上下文,避免信息泄露。

常见踩坑:❌ "把所有历史都放进 context window"--会爆,必须有压缩/截断策略。

🎯 从通用到落地：场景收窄八步法

TL;DR：你在 Part 1.5 搭了一个“通用 Agent Runtime”——它什么都能做，但什么都做得一般。要让它在具体场景下真正好用，需要“收窄”——把万能型变成专家型。这八步是产品级 Agent 调优的核心方法论。

💡 为什么需要“收窄”？
一个类比：你可以给一个新员工发一本《全公司工作手册》，但他第一天什么都做不好。更有效的做法是：只给他这个岗位的 SOP，限定他能用的工具，明确判断规则。

Agent 也一样：通用 System Prompt + 所有工具 = 什么都能做但什么都不大行。收窄后的 Agent = 专家。

八步法全览

  ① 评测集  →  ② 工具收窄  →  ③ 描述强化  →  ④ 策略规则
     │                                              │
     │         ⑤ Prompt 分层  ←  ④ 策略规则         │
     │              │                                │
     ▼              ▼                                ▼
  ⑥ 推理模式  →  ⑦ 消融实验  →  ⑧ 失败归因
                       │
                       ▼
                  回到 ① 迭代

① 评测集（Evaluation Set）—— 一切优化的起点

在改任何东西之前，先有一套可重复的测试用例。否则你不知道改了之后是变好了还是变差了。

# 评测集的最简形式：一组 (input, expected_output, score_func) 三元组
test_cases = [
    {"input": "北京明天天气怎么样",
     "expected": "包含温度和天气描述",
     "check": lambda out: "度" in out and len(out) > 20},
    {"input": "帮我算 15% 的小费，账单 86 块",
     "expected": "12.9",
     "check": lambda out: "12.9" in out},
]

评测集不需要很大——20-50 个 case 就能覆盖主要场景。关键是要包含正常情况、边缘情况和已知的失败情况。

② 工具收窄（Tool Narrowing）

不是给 Agent 所有工具，而是只给它这个场景需要的。工具越多，LLM 选错的概率越高。

# 场景：客服机器人，只需要 3 个工具
enabled_tools = ["query_order", "create_ticket", "check_refund_policy"]

# 而不是把 20 个工具全给它（搜索/计算器/发邮件/...）
# 工具越多，token 消耗越大，选错的概率也越高

③ 描述强化（Description Enhancement）

工具描述是单点 ROI 最高的优化——写清必填字段名、类型、枚举值，比改十次 System Prompt 有效。

# ✗ 差的描述
"description": "查询订单"

# ✓ 好的描述
"description": "根据订单号查询订单状态。输入: order_id(字符串，必填，格式如 'ORD-20240101-001')。返回: 订单状态(pending/shipped/delivered/cancelled)、金额、时间。如果用户没给订单号，先问用户要。"

④ 策略规则引擎（Policy Rules）

业务规则不要全堆在 System Prompt 里——做成“调用工具前注入的结构化提醒”。

# 示例：当 Agent 准备调用 create_refund 时，动态注入规则
policy_rules = {
    "create_refund": """⇨ 退款规则提醒：
    1. 超过 30 天不可退
    2. 已拆封的电子产品不可退
    3. 金额 > 500 元需人工审核
    请先检查这三个条件再决定是否执行。"""
}

# 在 before_tool_call 钩子里注入
if tool_name in policy_rules:
    messages.append({"role": "system", "content": policy_rules[tool_name]})

这就是 Ch03 提到的 before_tool_call 钩子的实际应用。规则只在需要的时候出现，不占永久 token。

⑤ Prompt 分层（Prompt Layering）

System Prompt 不应该是一块大杂烩，而应该分三层：

# 系统层：身份 + 红线（永远存在）
"你是一个电商客服助手。绝对不能泄露内部价格策略。"

# 场景层：当前任务上下文（按场景动态加载）
"当前用户是 VIP 客户，历史订单 12 笔，退款率 0%。"

# 执行层：当前步骤的具体指令（工具调用前注入）
"⇨ 退款规则提醒：超过 30 天不可退..."

⑥ 推理模式（Reasoning Mode）

不是所有任务都需要多步推理。简单问答用快模式，复杂决策才开慢模式（如 o1 / DeepSeek-R1 的 thinking）。

# 简单场景：“我的订单到哪了？” → 直接查工具，不需要思考
# 复杂场景：“我买的电脑有问题但已拆封，能退吗？” → 需要多步判断

# 可以用一个“分类器”先判断复杂度，再决定用哪个模式
complexity = classify(user_input)  # simple / complex
if complexity == "complex":
    # 用更强的模型 + 开启 thinking
    result = chat(messages, model="o1", reasoning=True)

⑦ 消融实验（Ablation Study）—— 关键！

每加一个机制，关掉再跑一次评测集，看分数掉多少。如果关掉了分数不变，那这个机制就是多余的——删掉它，减少复杂度和成本。

# 消融实验示例
baseline_score    = run_eval(all_features_on)    # 85%
no_policy_rules   = run_eval(policy_rules=False) # 72% → 掉 13%，必须保留！
no_tool_narrowing = run_eval(all_tools=True)     # 78% → 掉  7%，有用
no_reasoning      = run_eval(reasoning=False)   # 83% → 掉  2%，性价比低，可以去掉
💡 消融实验的心智模型
消融实验借鉴自神经科学——“想知道某个脑区的功能？把它损毁，看什么能力消失了。”同样，想知道你的某个优化是否有用？关掉它，看分数掉多少。

这是 Agent 调优中最被低估的方法——很多人加了一堆机制但从不验证哪个真正有用。结果系统越来越复杂，但很多复杂性是无用的。

⑧ 失败归因（Failure Attribution）

当评测集中的 case 失败时，不要盲目改 Prompt。先分类失败原因：

失败类型表现对策
工具选错该查订单却去算数学优化工具描述（步骤③）
参数错工具选对了但传错参数加 JSON Schema 约束 + 示例
规则违反超过30天的商品也给退款了加策略规则（步骤④）
推理错多步判断中逻辑出错开启推理模式（步骤⑥）
幻觉编造了不存在的订单信息回顾 Ch08 幻觉缓解策略

成本优先级：先砠什么？

不是所有优化都值得做。按 ROI 排序：

  ROI 从高到低：

  1️⃣ 评测集          —— 入场券，没有它其他都是猜
  2️⃣ 工具描述强化      —— 单点效果最明显，改一次管很久
  3️⃣ 策略规则注入      —— 解决业务逻辑错误
  4️⃣ Prompt 分层       —— 减少 token 消耗，提升相关性
  5️⃣ 推理模式         —— 只对复杂场景开，性价比低
  6️⃣ 换更强的模型      —— 放最后！上面 5 步做完再考虑
🎯 面试高频题
Q: 你们的 Agent 产品是怎么优化的？效果怎么评估？

标准答案：我们用八步收窄法——①先建评测集作为基线，②缩小工具范围减少选错，③强化工具描述（这是 ROI 最高的单点优化），④加策略规则引擎（工具调用前动态注入业务规则），⑤Prompt 分层（系统层/场景层/执行层），⑥复杂场景开推理模式，⑦每加一个机制做消融实验验证效果，⑧失败 case 分类归因。效果评估用评测集 + 人工抽样双重检查。

加分：提到成本优先级（先砠工具描述和规则，最后才换模型），以及消融实验的重要性（“不做消融实验的优化都是猜”）。

常见踩坑：❌ “我们调了很多次 Prompt”——没有评测集和消融实验的 Prompt 调优就是撞大运。
