🔨 Agent Hello World：从零搭一个能跑的 Agent Runtime

TL;DR：用 ~200 行 Python，从零搭一个完整可运行的 mini Agent。不是调 API 的脚本——是一个真正的 Runtime，有 ReAct 循环、工具注册、状态机、循环检测和可观测性。

💡 这一章的目标
Part 1 教了你 Agent 是什么（理论），这一章让你亲手搭一个（实践）。

读完这一章，你会有一个能在终端里跑的 Agent——它能接收你的问题，自己决定调什么工具，观察结果，再决定下一步，直到给你一个最终回答。

更重要的是：你会理解"一个 Agent Runtime 到底需要什么"。这样后面读 OpenClaw 源码时，每个模块你都知道它在干什么。

📐 架构设计：5 个文件，各司其职

在写代码之前，先想清楚要什么。一个最小 Agent Runtime 需要 5 个模块：

mini-agent/
├── types.py    ← 数据结构定义（Message、ToolCall、ToolResult）
├── llm.py      ← 统一 LLM 接口（对接 OpenAI/DeepSeek/硅基流动）
├── tools.py    ← 工具注册表 + 3 个真实工具
├── engine.py   ← ReAct 主循环 + 状态机 + 循环检测（核心！）
└── main.py     ← 入口，终端交互

为什么分 5 个文件而不是写一个？因为这正好映射了 Agent 的核心架构：

types.py  →  Ch03 的消息协议（tool_calls 后面必须跟 tool_result）
llm.py    →  Ch11 的 LLM API 调用（统一接口，可换模型）
tools.py  →  Ch03 的 Tool Use（注册 + 执行 + JSON Schema）
engine.py →  Ch02 的 ReAct 循环 + Ch01 的 LAMP 四组件
main.py   →  用户交互入口
💡 设计原则
关注点分离（Separation of Concerns）——每个文件只做一件事。这样你换模型只改 llm.py，加工具只改 tools.py，调循环策略只改 engine.py。

这不是过度设计——这是 Agent 系统的最小正确结构。后面你读 OpenClaw 源码会发现，它的模块划分和这个一模一样，只是每个模块复杂 100 倍。

Step 1 · types.py — 数据结构

先定义 Agent 内部流转的数据结构。就三个东西：消息（Message）、工具调用（ToolCall）、工具结果（ToolResult）。

# types.py — Agent 的数据类型定义
from dataclasses import dataclass, field
from typing import Any
from enum import Enum

# ── Agent 状态机 ──
class AgentState(Enum):
    IDLE    = "idle"      # 等待任务
    RUNNING = "running"   # 正在执行 ReAct 循环
    DONE    = "done"      # 任务完成
    ERROR   = "error"     # 出错了

# ── 工具调用请求（LLM 输出的）──
@dataclass
class ToolCall:
    id: str             # 调用 ID（用于匹配结果）
    name: str           # 工具名
    arguments: dict     # 参数

# ── 工具执行结果 ──
@dataclass
class ToolResult:
    tool_call_id: str   # 对应哪个 ToolCall
    content: str        # 执行结果（文本）
    success: bool = True
💡 为什么需要 tool_call_id？
LLM 可能一次输出多个工具调用（并行调用）。tool_call_id 用来匹配"这个结果是哪个调用的"。

协议约束：在 OpenAI 的消息格式中，每个 tool_calls 后面必须紧跟对应的 tool 消息。顺序错了模型会报错。这个约束在 Ch03 讲过——现在你在代码里亲眼看到了。

状态机为什么重要？没有状态机，你的 Agent 就是一个 while True 循环——不知道自己在干什么、什么时候该停。状态机让 Agent 的行为可预测、可监控：

  ┌──────┐    run()     ┌─────────┐
  │ IDLE │ ──────────→  │ RUNNING │
  └──────┘              └────┬────┘
                             │
                    ┌────────┴────────┐
                    │                 │
              正常完成            出错/超限
                    │                 │
               ┌────▼───┐       ┌────▼────┐
               │  DONE  │       │  ERROR  │
               └────────┘       └─────────┘

Step 2 · llm.py — 统一 LLM 接口

把 LLM 调用封装成一个函数。关键：传入工具列表，如果 LLM 决定调工具就返回 tool_calls，否则返回文本。

# llm.py — 统一 LLM 调用接口
import os, json
from openai import OpenAI
from types_ import ToolCall  # 注意：文件名 types.py 和标准库冲突，
                              # 实际项目中建议命名为 types_.py 或放在包里

# ── 初始化客户端（从环境变量读配置）──
client = OpenAI(
    api_key=os.getenv("OPENAI_API_KEY"),
    base_url=os.getenv("OPENAI_BASE_URL", "https://api.openai.com/v1"),
)
MODEL = os.getenv("OPENAI_MODEL", "gpt-4o-mini")

def chat(messages: list, tools: list | None = None) -> dict:
    """
    调用 LLM，返回标准化的结果。
    
    Returns:
        {
            "content": str | None,      # 文本回复
            "tool_calls": [ToolCall] | None,  # 工具调用请求
            "raw_message": object       # 原始消息（用于追加到历史）
        }
    """
    kwargs = {"model": MODEL, "messages": messages}
    if tools:
        kwargs["tools"] = tools

    resp = client.chat.completions.create(**kwargs)
    msg = resp.choices[0].message

    # 解析工具调用
    parsed_calls = None
    if msg.tool_calls:
        parsed_calls = [
            ToolCall(
                id=tc.id,
                name=tc.function.name,
                arguments=json.loads(tc.function.arguments),
            )
            for tc in msg.tool_calls
        ]

    return {
        "content": msg.content,
        "tool_calls": parsed_calls,
        "raw_message": msg,  # 保留原始对象，追加到 messages 时用
    }
💡 为什么要封装这一层？
换模型只改一行。改个环境变量就能从 GPT-4o 切到 DeepSeek。engine.py 完全不用动——它只依赖 chat() 的返回格式，不关心底层用什么模型。

这就是 Ch03 讲的"统一接口"思想。OpenClaw 的 llm.ts 做的也是同样的事情，只不过它还处理了 streaming、重试、token 计费等生产级需求。
⚠️ types.py 命名冲突
Python 标准库有一个 types 模块。如果你的文件叫 types.py，import 会冲突。两种解决方案：
1. 把文件命名为 types_.py（简单粗暴）
2. 把所有文件放在一个包（package）里，用相对导入

本教程用方案 1，代码里 from types_ import ToolCall。

Step 3 · tools.py — 工具注册表

三个真实工具——不是 mock，是真的能跑的：

# tools.py — 工具注册表 + 3 个真实工具
import json, math, datetime, urllib.request

# ══════════════════════════════════════════
# 工具 1: 天气查询（wttr.in 免费 API，无需 key）
# ══════════════════════════════════════════
def get_weather(city: str) -> str:
    """查询指定城市的实时天气"""
    try:
        url = f"https://wttr.in/{city}?format=%C+%t+%h+%w&lang=zh"
        req = urllib.request.Request(url, headers={"User-Agent": "mini-agent/1.0"})
        with urllib.request.urlopen(req, timeout=5) as resp:
            return f"{city} 天气: {resp.read().decode().strip()}"
    except Exception as e:
        return f"天气查询失败: {e}"

# ══════════════════════════════════════════
# 工具 2: 计算器（支持四则运算和常用数学函数）
# ══════════════════════════════════════════
def calculate(expression: str) -> str:
    """计算数学表达式，支持 +, -, *, /, **, sqrt, sin, cos, log"""
    try:
        # 只允许安全的数学操作
        allowed = {"sqrt": math.sqrt, "sin": math.sin, "cos": math.cos,
                   "log": math.log, "abs": abs, "pi": math.pi, "e": math.e}
        result = eval(expression, {"__builtins__": {}}, allowed)
        return f"{expression} = {result}"
    except Exception as e:
        return f"计算错误: {e}"

# ══════════════════════════════════════════
# 工具 3: 当前时间
# ══════════════════════════════════════════
def get_current_time() -> str:
    """获取当前日期和时间"""
    now = datetime.datetime.now()
    return f"当前时间: {now.strftime('%Y-%m-%d %H:%M:%S')} ({now.strftime('%A')})"

# ══════════════════════════════════════════
# 工具注册表 — 核心！
# ══════════════════════════════════════════
TOOL_REGISTRY = {
    "get_weather": get_weather,
    "calculate": calculate,
    "get_current_time": get_current_time,
}

# OpenAI Function Calling 格式的工具描述（JSON Schema）
TOOL_SCHEMAS = [
    {
        "type": "function",
        "function": {
            "name": "get_weather",
            "description": "查询指定城市的实时天气信息，包括温度、湿度、风力",
            "parameters": {
                "type": "object",
                "properties": {
                    "city": {
                        "type": "string",
                        "description": "城市名称，如 Beijing、Shanghai、Tokyo"
                    }
                },
                "required": ["city"]
            }
        }
    },
    {
        "type": "function",
        "function": {
            "name": "calculate",
            "description": "计算数学表达式。支持四则运算(+,-,*,/)、幂运算(**)、以及 sqrt/sin/cos/log 等函数。例如: '2**10' 或 'sqrt(144)'",
            "parameters": {
                "type": "object",
                "properties": {
                    "expression": {
                        "type": "string",
                        "description": "要计算的数学表达式"
                    }
                },
                "required": ["expression"]
            }
        }
    },
    {
        "type": "function",
        "function": {
            "name": "get_current_time",
            "description": "获取当前的日期、时间和星期几",
            "parameters": {
                "type": "object",
                "properties": {}
            }
        }
    },
]

def execute_tool(name: str, arguments: dict) -> str:
    """执行工具并返回结果"""
    func = TOOL_REGISTRY.get(name)
    if not func:
        return f"错误: 未知工具 '{name}'"
    try:
        return func(**arguments)
    except Exception as e:
        return f"工具执行失败: {e}"
💡 注册表模式（Registry Pattern）
TOOL_REGISTRY 是一个 名称 → 函数 的字典。要加新工具？只需要：1）写一个函数，2）加到 TOOL_REGISTRY，3）加一个 schema 到 TOOL_SCHEMAS。engine.py 完全不用改。

这就是 Ch03 讲的"工具注册"——MCP 做的也是同样的事情，只不过通过 JSON-RPC 协议远程注册，而不是 Python 字典。
🔧 工具描述的重要性
看 TOOL_SCHEMAS 里的 description 字段——写得越清楚，LLM 越能准确判断什么时候该调什么工具。

一个真实的坑：如果你把 calculate 的描述写成"计算"两个字，LLM 可能在不该调计算器的时候调它。描述里写清输入格式（"例如: '2**10'"）和能力边界（"支持四则运算和 sqrt/sin/cos/log"），LLM 的工具选择准确率会显著提高。

这就是团队 review 中提到的"工具描述强化"——在生产 Agent 中，优化工具描述的投入产出比往往高于改 System Prompt。

Step 4 · engine.py — ReAct 引擎（核心！）

这是整个 Agent 的心脏。它做的事情就是 Ch02 讲的 ReAct 循环：思考 → 行动 → 观察 → 思考 → ...直到完成。

# engine.py — ReAct 引擎：Agent 的心脏
import time
from types_ import AgentState, ToolCall
from llm import chat
from tools import TOOL_SCHEMAS, execute_tool

# ── 配置 ──
MAX_TURNS = 10          # 最多 10 轮（Ch02: 防无限循环）
MAX_REPEAT = 3          # 同一工具+参数连续重复 3 次则终止
SYSTEM_PROMPT = """你是一个有用的 AI 助手。
你可以使用工具来获取信息。先思考用户需要什么，再决定是否调用工具。
如果你已经有足够的信息来回答，就直接回答，不要多余地调用工具。"""

class Agent:
    def __init__(self):
        self.state = AgentState.IDLE
        self.messages = [{"role": "system", "content": SYSTEM_PROMPT}]
        self.turn = 0
        self.recent_calls = []  # 用于循环检测

    def run(self, user_input: str) -> str:
        """
        运行一次完整的 ReAct 循环。
        输入: 用户的问题
        输出: Agent 的最终回答
        """
        self.state = AgentState.RUNNING
        self.messages.append({"role": "user", "content": user_input})
        self.turn = 0
        self.recent_calls = []

        try:
            while self.state == AgentState.RUNNING:
                self.turn += 1

                # ── 安全检查 ──
                if self.turn > MAX_TURNS:
                    self._log("⚠️", f"达到最大轮次 ({MAX_TURNS})，强制终止")
                    self.state = AgentState.ERROR
                    return "抱歉，我在这个问题上花了太多步骤。请换个方式提问。"

                # ── Step 1: 调用 LLM（思考）──
                self._log("🧠", "思考中...")
                result = chat(self.messages, tools=TOOL_SCHEMAS)

                # ── Step 2: 判断 LLM 的决定 ──
                if result["tool_calls"]:
                    # LLM 决定调工具 → 行动
                    self._handle_tool_calls(result)
                else:
                    # LLM 给出最终回答 → 完成
                    answer = result["content"] or "(空回复)"
                    self.messages.append({"role": "assistant", "content": answer})
                    self._log("✅", f"最终回答: {answer[:100]}..." if len(answer) > 100 else f"最终回答: {answer}")
                    self.state = AgentState.DONE
                    return answer

        except Exception as e:
            self.state = AgentState.ERROR
            self._log("❌", f"出错: {e}")
            return f"Agent 出错了: {e}"

    def _handle_tool_calls(self, result: dict):
        """处理 LLM 返回的工具调用请求"""
        # 先把 LLM 的原始消息追加到历史（包含 tool_calls）
        self.messages.append(result["raw_message"])

        for tc in result["tool_calls"]:
            # ── 循环检测 ──
            call_sig = f"{tc.name}:{tc.arguments}"
            self.recent_calls.append(call_sig)
            if self.recent_calls[-MAX_REPEAT:] == [call_sig] * MAX_REPEAT:
                self._log("🔄", f"检测到重复调用 {tc.name}，终止循环")
                self.state = AgentState.ERROR
                # 仍需返回 tool result，否则消息格式会坏
                self.messages.append({
                    "role": "tool", "tool_call_id": tc.id,
                    "content": "[循环检测] 工具调用被终止"
                })
                return

            # ── 执行工具 ──
            self._log("🔧", f"调用工具: {tc.name}({tc.arguments})")
            tool_result = execute_tool(tc.name, tc.arguments)
            self._log("👁", f"观察: {tool_result[:100]}")

            # ⚡ 关键：tool 消息必须紧跟在 tool_calls 后面
            self.messages.append({
                "role": "tool",
                "tool_call_id": tc.id,
                "content": tool_result,
            })

    def _log(self, emoji: str, message: str):
        """可观测性：打印每一步的状态"""
        print(f"  Turn {self.turn} {emoji} {message}")

这段代码虽然只有 ~80 行，但它包含了 Agent 引擎的所有核心机制。让我们逐个对照前面学过的概念：

代码中的机制对应章节作用
while self.state == RUNNINGCh02 ReAct 循环持续循环直到完成
chat(messages, tools)Ch03 Tool CallingLLM 决定调不调工具
MAX_TURNS = 10Ch02 防无限循环安全上限
recent_calls + MAX_REPEATCh02 循环检测检测重复工具调用
AgentState 枚举Ch01 LAMP 组件状态机让行为可预测
_log() 方法Ch08 可观测性每轮打印思考/行动/观察
messages 数组Ch05 短期记忆对话历史就是工作记忆
SYSTEM_PROMPTCh04 System Prompt定义 Agent 的行为规则
⚠️ 最容易写错的地方
消息顺序。当 LLM 返回 tool_calls 时，你必须：
1. 先把包含 tool_calls 的 assistant 消息追加到 messages
2. 再追加每个 tool 消息（tool_call_id 必须匹配）

顺序搞反或者漏了 tool_call_id，API 会直接报错。这个约束是 Ch03 讲的"协议约束"的活生生例子。

Step 5 · main.py — 入口

# main.py — 终端交互入口
from engine import Agent

def main():
    print("🤖 Mini Agent 已启动！(输入 'quit' 退出)")
    print("   试试: '北京天气怎么样' 或 '2的10次方是多少' 或 '现在几点'")
    print()

    agent = Agent()

    while True:
        user_input = input("你: ").strip()
        if not user_input:
            continue
        if user_input.lower() in ("quit", "exit", "q"):
            print("👋 再见!")
            break

        print()
        answer = agent.run(user_input)
        print(f"\n🤖 Agent: {answer}\n")

if __name__ == "__main__":
    main()

就这么简单。main.py 只做一件事：读用户输入 → 交给 Agent → 打印结果。所有逻辑都在 engine.py 里。

  1. Turn 1 🧠 思考：用户问天气 → 需要查天气

  3. Turn 1 👁 观察：拿到结果
  4. Turn 2 🧠 思考：信息够了，可以回答
  5. Turn 2 ✅ 完成：生成最终回答

你: 帮我算一下，如果北京今天22度，那华氏温度是多少？

  Turn 1 🧠 思考中...
  Turn 1 🔧 调用工具: calculate({'expression': '22 * 9/5 + 32'})
  Turn 1 👁 观察: 22 * 9/5 + 32 = 71.6

  Turn 2 🧠 思考中...
  Turn 2 ✅ 最终回答: 22°C 换算成华氏温度是 71.6°F。

你: 现在几点了？

  Turn 1 🧠 思考中...
  Turn 1 🔧 调用工具: get_current_time({})
  Turn 1 👁 观察: 当前时间: 2026-04-15 21:30:45 (Tuesday)

  Turn 2 🧠 思考中...
  Turn 2 ✅ 最终回答: 现在是 2026年4月15日 周二 21:30。
💡 注意观察
Agent 会自己决定调什么工具。你问天气它调 get_weather，问计算它调 calculate，问时间它调 get_current_time。你没有写任何 if-else——LLM 读了工具描述后自己判断的。这就是 Ch01 说的"目标驱动执行"。

💥 故意搞坏它——理解 Agent 的边界

跑通了很开心，但更重要的是理解它什么时候会坏。试试这些：

实验 1：触发循环检测

你: 帮我查一下火星的天气

  Turn 1 🧠 思考中...
  Turn 1 🔧 调用工具: get_weather({'city': 'Mars'})
  Turn 1 👁 观察: 天气查询失败: HTTP Error 404

  Turn 2 🧠 思考中...
  Turn 2 🔧 调用工具: get_weather({'city': 'Mars'})
  Turn 2 👁 观察: 天气查询失败: HTTP Error 404

  Turn 3 🧠 思考中...
  Turn 3 🔄 检测到重复调用 get_weather，终止循环

循环检测生效了！如果没有这个机制，Agent 会一直重试到 MAX_TURNS，浪费 10 次 API 调用。

实验 2：去掉 System Prompt

把 engine.py 里的 SYSTEM_PROMPT 改成空字符串 ""，再问同样的问题。你会发现：

  • Agent 可能在不需要工具的时候也调工具
  • 回答风格变得不稳定
  • 有时候直接输出英文

这就是 Ch04 说的——System Prompt 定义了 Agent 的"灵魂"。没有它，Agent 就是一个没有方向的高智商。

实验 3：删掉工具描述里的 description

把 TOOL_SCHEMAS 里某个工具的 description 改成空字符串。你会发现 LLM 选错工具的概率大幅上升——问天气却调了计算器。

这验证了我们在 Step 3 说的：工具描述是工具选择准确率的最大杠杆。

🎯 面试高频题
Q: 你在实际项目中，Agent 出现过什么问题？怎么解决的？

标准答案：最常见的三个问题：1）无限循环——Agent 反复调用同一个失败的工具。解决：设 max_turns + 循环检测（连续 N 次相同调用则终止）。2）工具选择错误——LLM 选错工具或传错参数。解决：优化工具描述（加必填字段说明、示例值、能力边界），比改 System Prompt 有效。3）消息格式错误——tool_calls 后面没跟 tool result，或者 tool_call_id 不匹配，导致 API 报错。解决：严格按协议约束编码，tool result 必须紧跟 tool_calls 且 ID 匹配。

加分：这些问题我在自己搭的 mini Agent Runtime 里全都亲手复现过。
