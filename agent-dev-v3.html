01 · LLM 是怎么变成 Agent 的

TL;DR：Agent = LLM + Loop + Action(Tools) + Memory + Planning。LLM 是"大脑"，其余四个组件（LAMP 🪔）让它从"只能说"变成"能做事"。

直觉：从计算器到员工

想象你有一个超级聪明的朋友——他读过所有的书，能回答任何问题。但有个问题：他被关在一个房间里，只能通过纸条和你交流。

// 纯 LLM（被关在房间里的天才）
你写纸条：  "帮我订明天去上海的机票"
他回纸条：  "好的，你可以去携程网搜索..."  ← 他只能"说"，不能"做"

// Agent（放出来的天才，有手有脚有记忆）
你说：      "帮我订明天去上海的机票"
他：         打开携程 → 搜索航班 → 比价 → 下单 → 告诉你 "订好了，东航 MU5101，￥680"

这就是 LLM 和 Agent 的核心区别：LLM 是大脑，Agent 是一个完整的"人"。

Agent 的精确定义

AI Agent（AI 智能体）是一个能够自主感知环境、做出决策、并采取行动来实现目标的 AI 系统。

Agent 核心公式

Agent vs ChatBot vs Workflow vs RPA：深度对比

面试中经常会问这几个概念的区别。让我们用一个统一的场景来对比——"处理客户退款申请"：

灵活性只能处理预设场景只能走预定义路径UI 变了就挂能处理未知场景
退款场景"请问您的订单号是？" →

金额<100自动退→
否则转人工		打开后台→输入订单号→
点退款按钮→截图存档		理解用户诉求→查订单→

💡 一句话区别
ChatBot 是"会说话的菜单"，Workflow 是"自动化流水线"，RPA 是"模拟鼠标键盘的脚本"，Agent 是"能独立思考的员工"。

核心差异：前三者都需要人类预定义所有可能的路径，Agent 能自主决策面对未知情况。

Agent 的自主性（Autonomy）

Agent 的核心特征是目标驱动执行（Goal-driven Execution）——你给它一个目标，它自己决定怎么实现：

// 普通 LLM 调用：你决定每一步
response1 = llm.chat("搜索北京天气")     ← 你指挥
response2 = llm.chat("根据天气推荐穿搭") ← 你指挥
response3 = llm.chat("生成朋友圈文案")   ← 你指挥

// Agent：你只给目标，它自己决定怎么做
agent.run("帮我发一条关于今天天气的朋友圈")
  → Agent 自己决定：先查天气 → 再想穿搭 → 再写文案 → 发布
  → 如果天气 API 挂了：换一个 API 试试
  → 如果文案不够好：自己重写一遍

为什么 Agent 比 ChatBot 复杂得多？

ChatBot 的复杂度 = O(场景数量)
  // 每多一个场景，多写一组规则
  // 100 个场景 = 100 组规则

Agent 的复杂度 = O(能力组合 × 环境不确定性)
  // 10 个工具的组合方式 = 10! 种
  // 每步结果都可能偏离预期
  // 需要实时重新规划

这就是为什么 Agent 需要 LAMP 四个组件：
  Loop     → 持续运转，处理不确定性
  Tools    → 与外界交互的能力
  Memory   → 记住上下文和历史
  Planning → 拆解目标为可执行步骤
🎯 面试高频题
Q: 什么是 AI Agent？和普通 LLM API 调用有什么本质区别？

标准答案：AI Agent 是能自主感知环境、决策和行动的 AI 系统。核心组件是 LLM + Loop + Tools + Memory + Planning。与普通 LLM 调用的区别在于：LLM 调用是"一问一答"，人驱动每一步；Agent 是"给目标自己跑"，有自主循环、能调用工具、能记住上下文、能拆解任务。

加分回答：可以用 LAMP 记忆法解释四个组件，并举例说明——比如 ChatGPT 普通对话 vs Claude Code 自动修 bug 的区别。

常见踩坑：❌ "Agent 就是加了工具的 LLM"——这只说了 1/4，漏了 Loop、Memory 和 Planning。
🎯 面试高频题
Q: Agent 和 Workflow 的区别是什么？

标准答案：Workflow 是预定义的确定性流程（if-else 分支），所有路径在设计时就固定了。Agent 是非确定性的，它根据实时观察结果动态决策下一步。Workflow 适合流程稳定的场景，Agent 适合需要灵活应对的场景。

加分回答：实际系统中两者常结合——用 Workflow 做确定性骨架，在不确定的节点嵌入 Agent 做决策。比如退款流程用 Workflow，但"判断退款是否合理"这一步用 Agent。

常见踩坑：❌ "Agent 可以完全替代 Workflow"——不是。对于规则明确的流程，Workflow 更可靠、更便宜、更可审计。
🎯 面试高频题
Q: 模型和 Agent 的区别是什么？

标准答案：模型（LLM）是 Agent 的"大脑"——一个输入文本、输出文本的数学函数。Agent 是包含模型在内的完整系统，还包括工具调用、记忆管理、任务规划和执行循环。模型本身不能操作外部世界，Agent 可以。

加分回答：类比：模型 = CPU，Agent = 整台电脑（CPU + 硬盘 + 网卡 + 操作系统）。CPU 再强也不能单独上网，得有完整系统才行。

02 · ReAct 循环：思考-行动-观察

TL;DR：ReAct = Reasoning + Acting。Agent 的每一步都是"先想再做，看结果，再想"——不断循环直到任务完成。这是 Agent 的心跳。

直觉：你去陌生城市找餐厅

🧠 思考：我饿了，想吃火锅，先查一下附近有什么
🔧 行动：打开大众点评搜索"附近火锅"
👁 观察：海底捞 4.8分，排队 2小时

🧠 思考：太久了，看第二家
🔧 行动：点开第二家
👁 观察：评分 4.6，不用排队，步行 10 分钟

🧠 思考：可以，走吧
🔧 行动：导航过去
✅ 完成：开吃！

你没有提前规划好所有步骤——你是边做边想，根据每步结果决定下一步。这就是 ReAct。

ReAct 循环

三种方法对比

方法过程缺点
Chain-of-Thought
只想不做		LLM 纯推理，不调用工具可能在"瞎猜"，没有真实数据
纯 Tool Calling
只做不想		收到请求就调工具，不推理不会反思结果，不会换方案
ReAct ✅
边想边做		每步先推理→再决定行动→看结果→再推理可能比较慢（多轮 LLM 调用）

ReAct 的变体模式

面试会问"ReAct 那几个模式有什么差别"。主要有三种：

1. 标准 ReAct
   思考 → 行动 → 观察 → 思考 → ...
   每一步都"想"，适合需要深度推理的任务

2. Act-only（纯行动）
   行动 → 观察 → 行动 → 观察 → ...
   省略思考步骤，靠工具结果直接驱动
   快但容易跑偏，适合简单任务

3. Plan-then-Execute
   规划 → [行动 → 观察] × N
   先做完整计划，再按计划执行
   适合步骤明确的任务，但不够灵活

对比：
标准 ReAct  = 边走边看路的司机（灵活但可能绕路）
Act-only   = 闷头开的司机（快但可能撞墙）
Plan-first = 先看完地图再出发（稳但不能应对堵车）

如何避免无限循环？

这是面试高频问题。Agent 陷入死循环是生产中最常见的问题之一：

// 死循环示例
🧠 思考：搜索结果不对，换个关键词再搜
🔧 行动：web_search("xxx")
👁 观察：还是找不到
🧠 思考：再换一个关键词...
🔧 行动：web_search("yyy")
👁 观察：还是找不到
→ 无限循环...

// 四种防护策略
1. 最大轮次限制（Max Iterations）
   max_turns = 25  // 超过就强制停止

2. 超时机制（Timeout）
   timeout = 300  // 5 分钟超时

3. 重复检测（Loop Detection）
   如果连续 3 次调用同一个工具且参数相似 → 强制终止

4. Token 预算（Budget）
   max_tokens = 100000  // 花完就停

Checkpoint / Resume：中断恢复

// Agent 在执行长任务时，每完成一步保存状态
{
  task: "分析 10 个竞品",
  checkpoint: {
    completed: ["竞品A", "竞品B", "竞品C"],
    current: "竞品D",
    remaining: ["竞品E", ... "竞品J"],
    partial_results: { ... }
  }
}

// 如果中断（超时/崩溃），可以从 checkpoint 恢复
agent.resume(checkpoint)  → 从"竞品D"继续，不用从头来
🎯 面试高频题
Q: 什么是 ReAct？Agent 如何避免无限循环调用工具？

标准答案：ReAct = Reasoning + Acting，Agent 交替进行推理和行动：先思考下一步做什么→执行→观察结果→再思考。避免无限循环的策略：①设置最大轮次（如 25 轮）②设置超时（如 5 分钟）③重复调用检测（连续相同工具调用则终止）④Token 预算限制。

加分回答：提到 ReAct 的三种变体（标准/Act-only/Plan-then-Execute）的区别和适用场景。提到 OpenClaw 的 Agent Loop 默认超时是 48 小时，通过 per-session + global queue 序列化来保证安全。

常见踩坑：❌ "设一个 max_turns 就够了"——不够，需要多层防护；❌ 把 ReAct 和 React（前端框架）搞混。

03 · Tool Use / MCP / Skill：给 Agent 接入世界的三种方式

TL;DR：LLM 不能执行工具——它只是"说"想调用什么。真正执行的是外部系统。Tool Calling 是基础协议，MCP 是工具连接标准，Skill 是行为指令标准，A2A 是 Agent 间通信标准。

直觉：LLM 是老板，Tool 是员工

老板（LLM）："小王，帮我查一下北京今天天气"
                  ↓
  老板不会自己查——他写了一张便条：
  { "tool": "get_weather", "args": { "city": "北京" } }
                  ↓
  秘书（Runtime）把便条交给气象组（Tool）
                  ↓
  气象组执行查询，返回结果："22°C，晴"
                  ↓
  秘书把结果递给老板
                  ↓
  老板看了结果，继续下一步决策

Function Calling 协议详解

当你调用 OpenAI/Anthropic 的 API 时，工具调用实际上是这样工作的：

// Step 1: 你告诉 LLM 有哪些工具可用
{
  "model": "gpt-4",
  "messages": [{ "role": "user", "content": "北京天气怎么样？" }],
  "tools": [{
    "type": "function",
    "function": {
      "name": "get_weather",
      "description": "获取指定城市的天气信息",
      "parameters": {
        "type": "object",
        "properties": {
          "city": { "type": "string", "description": "城市名" }
        }
      }
    }
  }]
}

// Step 2: LLM 返回"我想调用这个工具"（不是直接执行！）
{
  "role": "assistant",
  "tool_calls": [{
    "id": "call_abc123",
    "function": { "name": "get_weather", "arguments": "{\"city\":\"北京\"}" }
  }]
}

// Step 3: 你的代码执行工具，把结果送回去
{
  "role": "tool",
  "tool_call_id": "call_abc123",
  "content": "北京今天 22°C，晴，微风"
}

// Step 4: LLM 根据工具结果生成最终回答

┌──────────────────────────────────────┐
│  Host（宿主）                         │
│  IDE / Chat UI / Agent Orchestrator  │
│  └─ 内嵌一个或多个 MCP Client          │
└──────────┬───────────────────────────┘
           │ JSON-RPC（有状态会话）
┌──────────▼───────────────────────────┐
│  MCP Server（服务端）                  │
│  暴露三种原语（Primitives）：           │

│                                       │

// MCP 的生命周期
1. Client 发送 initialize → 协商支持的功能（capabilities）
2. Client 调用 tools/list  → 发现可用工具
3. Client 调用 tools/call  → 执行某个工具
4. Server 发送通知        → 如工具列表变更
5. Client 发送 shutdown   → 关闭会话

// 关键设计选择：
// • 有状态会话（不是无状态 REST）
// • JSON-RPC 2.0 作为消息格式
// • 服务端可以主动推送通知
// • 一个 Host 可以连多个 MCP Server

Skill 和 MCP 的区别

// MCP 给 Agent 的是"能力"（access）
MCP: "你可以调用 get_weather、send_email、query_db..."

// Skill 给 Agent 的是"纪律"（discipline）
Skill: "当用户问天气时，先查城市→再查天气→用友好格式回复→
        如果查不到就推荐最近的城市→回复要包含穿衣建议"

// Skill 通常建立在 MCP 工具之上
// MCP = 给你一把刀（工具）
// Skill = 教你怎么用这把刀切菜（方法论）

工具调用失败：降级策略

// 生产环境中，工具调用会失败。必须有降级方案：

策略 1: 重试（Retry）
  tool_call → 失败 → 等 1s → 重试 → 成功
  适合：网络抖动、临时错误

策略 2: 备用工具（Fallback Tool）
  weather_api_A → 失败 → 尝试 weather_api_B
  适合：有多个同类工具可选

策略 3: 优雅降级（Graceful Degradation）
  精确天气 → 失败 → 返回"近期北京天气一般在 15-25°C"
  适合：有历史数据或通用知识可用

策略 4: 跳过并告知（Skip & Inform）
  "抱歉，天气查询暂时不可用，我先帮你完成其他部分"
  适合：非关键步骤

策略 5: 人工介入（Human-in-the-Loop）
  "这步需要你手动确认..."
  适合：高风险操作

结构化输出

// 问题：LLM 输出的是自然语言，怎么得到结构化 JSON？

方法 1: Prompt 引导（最简单）

Q: MCP 是什么？本地工具、MCP Tools、Skill 三者的区别？

标准答案：MCP（Model Context Protocol）是 Anthropic 提出的开放标准，标准化 AI 应用连接外部工具/数据的方式，采用 JSON-RPC 协议。三者区别：
• 本地工具：直接在代码中定义的函数，硬编码在 Agent 里
• MCP Tools：通过 MCP 协议远程连接的标准化工具，可插拔、跨平台
• Skill：不是工具本身，而是"怎么使用工具"的行为指令（SKILL.md），通常建立在 MCP 工具之上

加分回答：解释为什么 Anthropic 在有了 MCP 之后还做 Skill——因为 MCP 解决的是"连接"问题（access），Skill 解决的是"应用"问题（discipline）。一个给能力，一个给方法论。再加上 Google 的 A2A 解决"Agent 间通信"，三者互补。

常见踩坑：❌ "Skill 是 MCP 的替代品"——不是，它们在不同层次；❌ 不知道 MCP 用 JSON-RPC 协议。
🎯 面试高频题
Q: 工具调用失败怎么办？降级方案怎么设计？

标准答案：五层降级：①重试（网络抖动）②备用工具（同类替换）③优雅降级（用缓存/通用知识替代）④跳过非关键步骤⑤人工介入。生产中通常配合超时设置（如 30 秒）和熔断器模式（连续失败 N 次则暂停该工具）。

加分回答：提到 OpenClaw 的做法——通过 Harness Engineering 的"熵管理"来控制工具失败的连锁反应，以及 before_tool_call / after_tool_call 钩子可以拦截和处理失败。

常见踩坑：❌ 只说"设置重试"——重试不能解决所有问题，需要多层策略。

04 · 系统提示词：Agent 的"灵魂"

TL;DR：System Prompt 定义了 Agent "是谁"和"怎么行为"。好的 Prompt 设计是 Agent 开发中投入产出比最高的工作。

三种 Role 的作用

// LLM API 中的三种角色
messages = [
  { role: "system",    content: "你是一个天气助手..." },  ← 定义身份和规则
  { role: "user",      content: "北京天气怎么样" },    ← 用户的输入
  { role: "assistant", content: "北京今天22°C..." },   ← AI 的回复（历史）

System：人格 + 规则 + 约束（最重要，每次都会注入）

更新策略：

LRU 原理与实现

// LRU = Least Recently Used（最近最少使用）
// 用于限制记忆容量：满了就淘汰最久没用的

class LRUCache {
  constructor(capacity) {
    this.capacity = capacity;
    this.map = new Map();  // Map 保持插入顺序
  }

  get(key) {
    if (!this.map.has(key)) return undefined;
    const value = this.map.get(key);
    this.map.delete(key);     // 删除
    this.map.set(key, value);  // 重新插入（变成最新）
    return value;
  }

  put(key, value) {
    if (this.map.has(key)) this.map.delete(key);
    this.map.set(key, value);
    if (this.map.size > this.capacity) {
      // 淘汰最老的（Map 的第一个）
      const oldest = this.map.keys().next().value;
      this.map.delete(oldest);
    }
  }
}

// 为什么需要 Rerank？
// 初始检索（embedding 向量匹配）是"粗筛"——快但不够精
// Rerank 是"精排"——慢但更准

        ↓
Rerank 模型（cross-encoder）→ 重新打分排序
        ↓
取 Top 3-5 → 送给 LLM 生成

// 为什么 Rerank 比 embedding 更准？
Embedding（Bi-Encoder）：  问题和文档分别编码，用余弦比较
  快（可以预计算），但精度有限

Rerank（Cross-Encoder）：  问题+文档拼在一起，同时编码
  慢（不能预计算），但精度更高
  因为它能看到问题和文档之间的细粒度交互

Q: 一个标准的 RAG 系统包含哪些模块？完整流程是什么？

标准答案：分两个阶段——

🎯 面试高频题
Q: Chunk size 怎么选？为什么很重要？

标准答案：Chunk size 直接影响检索质量。太小（如 100 token）：检索精准但缺上下文，LLM 没有足够信息生成好的回答。太大（如 2000 token）：上下文完整但检索不精准，且占用更多 context window。通常 256-512 token 是好的起点，需要根据具体数据和任务调优。

加分回答：提到父子索引（小块检索、返回大块），以及 overlap（chunk 之间有 10-20% 的重叠，避免关键信息被切断）。

常见踩坑：❌ 给一个固定数值不解释——chunk size 没有万能值，取决于文档类型和任务。

Q: Claude Code 的记忆架构是什么？为什么不用 RAG 检索代码而是用 grep？

标准答案：Claude Code 的记忆分三层：①Context Window（当前对话）②CLAUDE.md 文件（项目级指令）③Memory 文件（持久化笔记）。它用 grep/ripgrep 而非 RAG 搜索代码，因为：代码搜索通常需要精确匹配（函数名、变量名），grep 在这方面比向量检索更准更快。向量检索擅长语义匹配（"类似的意思"），但代码中需要的是"精确的符号匹配"。

加分回答：补充说"上下文≠记忆"——context window 是临时的，每次对话结束就没了；真正的记忆需要持久化到文件系统（OpenClaw 用 MEMORY.md + memory/*.md）。

常见踩坑：❌ "RAG 比 grep 更高级所以应该用 RAG"——选工具要看场景，精确匹配场景 grep 更优。

🔥 幻觉（Hallucination）详解

// 幻觉 = LLM 生成了看起来正确但实际错误的内容
// 研究表明：领先 AI 模型在科学摘要中幻觉率高达 31%

两大类型（最新分类法）：
① 知识型幻觉（Knowledge-based）
   → 事实错误："爱因斯坦于 1950 年获诺贝尔奖"（错，是 1921）
   → 引用虚构不存在的论文/链接
   → 原因：训练数据缺失或矛盾

② 逻辑型幻觉（Logic-based）
   → 推理过程看起来对但结论错
   → 多步推理中错误累积
   → 原因：自回归模型的"模式延续"倾向

根本成因（从架构角度）：
① 自回归生成 — 模型预测下一个 token 时倾向于"自信补全"
② 知识缺口 — 对长尾知识或最新信息覆盖不足
③ 模式驱动 — 生成了语法正确但事实错误的续写
④ Context 稀释 — 长上下文中 Attention 衰减（Lost in the Middle）
⑤ 训练数据噪声 — 互联网数据本身包含错误信息

7 层缓解策略（从轻到重）：
① Prompt 工程     — 明确指令："只基于给定信息回答，不确定则说不知道"
② RAG             — 检索真实文档作为上下文（最有效的单一策略）
③ Grounding       — 要求模型引用来源，每句话标注出处
④ Self-check      — 生成后让模型自己验证（"请检查上述回答是否有事实错误"）
⑤ Constrained 输出 — 结构化输出 + 限制回答范围
⑥ 外部验证       — 自动化 fact-checker 验证关键事实
⑦ Fine-tuning    — 用领域专家标注的数据微调模型

// 实际生产中通常组合使用：RAG + Grounding + Self-check

Q: 什么是幻觉（Hallucination）？为什么会发生？如何减少？

标准答案：幻觉是 LLM 生成看似正确但实际错误的内容。成因：①训练数据噪声②模型倾向于补全而非拒绝③上下文信息不足。缓解：①RAG（提供真实参考文档）②要求引用来源③Self-consistency check④降低 temperature⑤结构化输出约束。

加分回答：区分三种幻觉类型（事实/逻辑/引用），并提到 RAG 中的 Faithfulness 指标——评估生成内容是否忠于检索文档。

常见踩坑：❌ "用 RAG 就能完全消除幻觉"——不能，但能大幅减少；RAG 本身也可能检索到错误文档。

手写 RAG Pipeline（Python）

from openai import OpenAI
import numpy as np

client = OpenAI()

# ── 1. 准备文档并切分 ──
documents = [
    "OpenClaw 的 Gateway 默认端口是 18789",
    "OpenClaw 用 WebSocket 协议通信，不是 HTTP",
    "Agent 的 System Prompt 包含 Tools、Skills、Bootstrap 文件",
    "Session 会在每天凌晨 4 点自动重置",
]

# ── 2. 文本 → 向量 ──
def embed(texts):
    resp = client.embeddings.create(model="text-embedding-3-small", input=texts)
    return [r.embedding for r in resp.data]

doc_vectors = embed(documents)

# ── 3. 检索 ──
def search(query, top_k=2):
    q_vec = embed([query])[0]
    scores = [np.dot(q_vec, dv) for dv in doc_vectors]
    top_idx = np.argsort(scores)[-top_k:][::-1]
    return [documents[i] for i in top_idx]

# ── 4. RAG 生成 ──
question = "OpenClaw 用什么协议？"
context = search(question)

answer = client.chat.completions.create(
    model="gpt-4",
    messages=[
        {"role": "system", "content": f"基于以下参考文档回答：\n{chr(10).join(context)}"},
        {"role": "user", "content": question}
    ]
)
print(answer.choices[0].message.content)
# → "OpenClaw 使用 WebSocket 协议通信"

训练基础：SFT / RLHF / DPO / GRPO

// 面试中经常问训练流程，不需要太深但要清楚区别

SFT（Supervised Fine-Tuning，监督微调）
  数据：(instruction, response) 对
  方法：标准交叉熵损失，教模型"照着好答案学"
  类比：老师给标准答案，学生照着抄

RLHF（Reinforcement Learning from Human Feedback）
  Step 1: 训练一个 Reward Model（人类偏好打分器）
  Step 2: 用 PPO 算法优化 LLM，最大化 Reward
  类比：先教裁判什么是好答案，再让学生根据裁判反馈改进

DPO（Direct Preference Optimization）
  数据：(prompt, good_response, bad_response) 三元组
  方法：直接优化模型偏好，不需要 Reward Model
  类比：直接告诉学生"A比B好"，不需要裁判
  优点：比 RLHF 简单很多，训练更稳定

GRPO（Group Relative Policy Optimization）
  DeepSeek 提出，PPO 的简化版
  不需要 Reward Model，用组内样本相对排名做优化
  比 DPO 更灵活，比 PPO 更简单

加分回答：提到 GRPO（DeepSeek 提出），是 PPO 的简化版——不需要 Reward Model，用组内样本排名替代。

12 · Attention 机制 + Context Window

TL;DR：Attention 是 Transformer 的核心——让模型在处理每个词时能"关注"到所有相关的词。Context Window 是模型能"看到"的最大范围。

Attention 的本质（面试必问！）

用图书馆的类比来理解：

// 你走进图书馆（Context Window），要找和你问题相关的书

Q（Query） = 你的问题："什么是 Agent？"
K（Key）   = 每本书的标签/关键词
V（Value） = 每本书的内容

// Attention 的计算：
Step 1: 拿你的问题（Q）和每本书的标签（K）对比
        → 算出每本书的"相关度分数"（注意力权重）
        → score = Q · K^T / √d  (点积 / 缩放)

Step 2: 用 softmax 把分数归一化成概率
        → [0.05, 0.60, 0.30, 0.05, ...]
        （第二本书最相关）

Step 3: 按概率加权求和所有书的内容（V）
        → output = softmax(QK^T/√d) × V
        （主要是第二本书的内容 + 少量其他书的补充）
Self-Attention 计算流程

长上下文中 Attention 的衰减问题

// 问题：context window 越长，早期信息的 attention 权重越小
// 这就是"Lost in the Middle"现象

位置:  [开头] ────────── [中间] ────────── [结尾]
权重:  ████              ██                ████████

开头（system prompt）→ 权重高（位置编码优势）
中间（旧的对话历史）→ 权重低（最容易被"遗忘"）
结尾（最近的消息）  → 权重高（近因效应）

// 这就是为什么多轮对话越长，Agent 越容易"忘记"中间说过的话
// 解决方案：Compaction（把旧对话压缩成摘要放在前面）
🎯 面试高频题
Q: Attention 本质是什么？从向量空间变换角度解释。

标准答案：Attention 本质是一个加权信息聚合机制。每个 token 通过 Q/K/V 三个线性变换映射到不同的向量空间：Q（查询）代表"我在找什么"，K（键）代表"我有什么"，V（值）代表"我的内容"。通过 Q·K^T 计算相关度，softmax 归一化后加权求和 V，得到融合了全局信息的表示。数学上：Attention(Q,K,V) = softmax(QK^T/√d_k)V。

加分回答：提到 Multi-Head Attention（多个 head 关注不同方面，如语法/语义/指代），以及 √d_k 缩放是为了防止 softmax 在高维时梯度消失。

常见踩坑：❌ 只说"Attention 让模型关注重要的部分"——太模糊，要说出 Q/K/V 的计算过程。
