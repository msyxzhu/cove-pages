0A · 从人脑到人工神经网络

TL;DR：神经网络（Neural Network）是一种模仿大脑工作方式的计算模型。它由大量简单的"神经元"组成，通过调整连接权重来学习。训练 = 给它看大量例子，让它自己找到规律。

故事：让机器"看懂"手写数字

1986 年，邮局面临一个问题：每天有几百万封信，需要人工辨认手写的邮政编码。一个数字"7"，有人写得像"1"，有人画一横穿过去，有人写得歪歪扭扭。能不能让机器来做这件事？

传统程序员会尝试写规则："如果有一条竖线和一条横线在顶部交叉，那就是 7"。但人类手写的变化太多了，规则写不完。

这时候有人想到：别告诉机器规则，让机器自己从例子中学。给它几万张手写数字图片，每张标注正确答案，让它自己找到"7 长什么样"的规律。这就是神经网络的核心思想。

直觉：一群投票者

想象一个由 100 个人组成的评审团，要判断一张手写图片是不是数字"7"：

  • 每个评审员负责看图片的不同特征（有的看顶部横线，有的看右侧竖线，有的看整体形状）
  • 每个评审员投票："我觉得是 7"或"我觉得不是 7"
  • 但不是一人一票——有些评审员更靠谱，他们的票权重更大
  • 最终按加权投票结果做决定

训练的过程就是：不断调整每个评审员的权重。如果系统判断错了，那些投错票的评审员权重降低，投对的升高。重复几百万次后，这个评审团就变得非常准确了。

训练：不断试错的过程

神经网络的训练分两步，像考试和订正的循环：

  1. 前向传播（Forward Pass）— "做题"：数据从输入层流向输出层，每一层把输入乘以权重（Weight）再加上偏置（Bias），最终得到一个预测结果。
  2. 反向传播（Backpropagation）— "对答案 + 订正"：把预测结果和正确答案对比，计算差距（Loss），然后从后往前调整每一层的权重，让下次预测更准。

这两步不断重复。一个训练可能要循环几百万次，每次微调一点点权重。最终，神经网络从一个"瞎猜"的状态变成一个"很准"的状态。

图 0A-2：训练 = 不断缩小预测与真相的差距
💡 关键类比
训练神经网络就像教一个婴儿认动物：你拿着图片说"这是猫"、"这是狗"，重复几千次后，婴儿不是记住了每张具体的图，而是学会了"猫的特征是什么"这个模式（Pattern）。神经网络也一样——它学到的是数据中的模式，不是数据本身。
⚠️ 常见误区
"神经网络就像人脑一样思考" — 不对。虽然受大脑启发，但它的工作方式其实就是大量的加法和乘法运算。它不"理解"任何东西，只是找到了输入和输出之间的统计规律。这个区别在理解 LLM 时非常重要。

深入：损失函数与梯度下降——机器怎么"调整"权重？

前面我们反复说"不断调整权重让预测更准"——但机器到底怎么知道该把哪个权重调大、哪个调小？这就是训练的核心机制。

损失函数（Loss Function）——给"错误"打分

训练的第一步是量化"模型有多差"。损失函数就是这个度量工具：它计算模型预测结果和真实答案之间的差距，输出一个数字。

类比：考试分数的"反面"。考试分数越高越好，而损失值越低越好——损失为 0 意味着完美预测。

常见的损失函数：

  • 均方误差（MSE）：预测值和真实值差的平方再平均，常用于回归任务（预测数字）
  • 交叉熵损失（Cross-Entropy Loss）：衡量两个概率分布的差异，常用于分类任务。先记住这个名字——后面 0D 章讲 SFT 训练 LLM 时，用的就是交叉熵损失。
// 一个直觉例子
模型预测："这张图是猫的概率 = 0.3"
真实答案："这张图是猫"（概率 = 1.0）

损失 = 差距很大 → 损失值高 → 需要调整权重！

// 训练若干轮后...
模型预测："这张图是猫的概率 = 0.95"
真实答案："这张图是猫"（概率 = 1.0）

损失 = 差距很小 → 损失值低 → 快学会了！

梯度下降（Gradient Descent）——蒙眼下山

知道了"有多差"（损失值），下一步是"怎么变好"。这就是梯度下降。

经典类比：蒙眼登山者。想象你站在一座山上，周围浓雾弥漫，完全看不见山谷在哪。但你能做一件事——感受脚下的坡度。你伸脚探一探，找到最陡的下坡方向，迈出一步。重复这个过程，你最终会走到山谷（最低点 = 最小损失）。

  • 梯度（Gradient）= 脚下坡度的方向。数学上是损失函数对每个权重的偏导数，告诉你"这个权重增大会让损失变大还是变小"
  • 每一步= 一次参数更新。沿着梯度反方向调整所有权重
  • 步长 = 学习率（Learning Rate）= 每一步迈多大

💡 一句话总结训练
前向传播 = 做题，损失函数 = 打分，梯度下降 = 看错题本找方向，反向传播 = 按方向改答案。这四步循环就是神经网络训练的全部。

监督学习 vs 自监督学习

训练需要数据，但数据的"答案"从哪来？这取决于学习方式：

  • 监督学习（Supervised Learning）：人工标注了答案。比如给每张图片标了"这是猫""这是狗"，模型对照答案学习。传统机器学习和后面的 SFT（有监督微调）用的就是这种。
  • 自监督学习（Self-supervised Learning）：数据自己生成标签，不需要人工标注。比如"遮住句子中的一个词让模型猜"（BERT 的做法），或者"给前半句预测后半句"（GPT 的做法）。

这个区别很重要——后面 0D 章你会看到：Pre-training 用的是自监督学习（从海量文本自动生成训练数据），而 SFT 用的是监督学习（需要人工编写的高质量问答对）。

💡 为什么这些基础很重要
理解了损失函数和梯度下降，后面 0D 章讲 SFT 用"交叉熵损失"、RLHF 用"PPO 优化"时就不会懵了。这些不是新概念——只是同样的训练循环在不同场景下的应用。

0B · 语言的数字化——Tokenization 与 Embedding

上一章我们学了神经网络——一种能从数据中学习的"大脑"。但这个大脑有个限制：它只能处理数字。文字、语言，对它来说就像天书。那怎么办呢？

TL;DR：Tokenization（分词）把文本拆成小块（Token），Embedding（嵌入）把每个 Token 变成一组数字（向量）。意思相近的词，向量也相近。这是让机器"读懂"语言的第一步。

故事：机器怎么"读"一句话？

假设你要教一个只懂数学的外星人理解中文。你不能直接把汉字给它——它看不懂。你需要先把文字翻译成数字。

这个"翻译"分两步：

第一步：Tokenization（分词）——把文本拆成积木块

就像把一段文字拆成乐高积木。不是按字拆，也不一定按词拆，而是按"子词（subword）"拆——一种介于字和词之间的单位：

// Tokenization 示例（GPT 系列使用 BPE 算法）

"unhappiness" → ["un", "happi", "ness"]     ← 3 个 token
"ChatGPT"     → ["Chat", "G", "PT"]         ← 3 个 token
"你好世界"     → ["你好", "世界"]              ← 2 个 token
"我喜欢吃火锅" → ["我", "喜欢", "吃", "火锅"]  ← 4 个 token

// 常见模型的词汇表大小：
// GPT-4:  ~100,000 个 token
// Claude:  ~100,000 个 token
// Llama:   ~32,000 个 token
💡 为什么不按字/词拆？
按字拆：序列太长，计算成本高。按词拆：词汇表太大（英语有几十万个词），而且遇到新词就不认识了。子词（subword）是最佳平衡——词汇表大小适中，几乎能覆盖所有可能的文本组合。比如 "unhappiness" 从来没出现在训练数据中也没关系，因为 "un"、"happi"、"ness" 都见过。

第二步：Embedding（嵌入）——给每个 Token 一个 GPS 坐标

Tokenization 把文字变成了编号（Token ID），但编号本身没有含义——"猫"是 15234，"狗"是 8821，这两个数字的大小关系毫无意义。

但首先，什么是"向量"（Vector）？向量就是一组有序的数字。你每天都在用向量——GPS 坐标 [纬度, 经度] 就是一个 2 维向量，RGB 颜色 [255, 128, 0] 就是一个 3 维向量。Embedding 就是给每个词一个"几百维的 GPS 坐标"，只不过维度多到人类无法直观想象。

Embedding 的作用是：把每个 Token 映射到一个高维空间中的坐标点。在这个空间里，意思相近的词距离近，意思不同的词距离远。

图 0B-1：词向量空间（2D 简化版）

最著名的 Embedding 发现：国王 - 男人 + 女人 ≈ 女王。这说明 Embedding 空间真的捕捉到了语义关系！向量的方向代表了"性别"、"皇室"等抽象概念。

// GPT-2:    768 维
// GPT-3:    12288 维
// GPT-4:    未公开，推测 ~12288 维

// "猫" 的 embedding 可能是这样的（简化到5维）：

🧩 和 Agent 的关系
以后学到 RAG（检索增强生成）时会发现，Embedding 是 RAG 的核心技术。Agent 要从大量文档中找到和用户问题最相关的片段，就是把问题和文档都变成 Embedding 向量，然后找距离最近的。
⚠️ Token ≠ 词
初学者最常犯的错：以为一个 Token 就是一个词。实际上一个英文词平均 ≈ 1.3 个 Token，一个中文字 ≈ 1.5-2 个 Token。当你看到 "Context Window: 128K tokens" 时，这不是 128K 个词——中文大约相当于 6-8 万字。理解这个换算关系在估算 API 成本和上下文管理时非常重要。

0C · Attention 机制——Transformer 的灵魂

上一章我们把文字变成了数字向量。现在每个词都有了一个"坐标"。但有个问题：这些坐标是静态的——"bank"无论出现在"river bank"（河岸）还是"bank account"（银行账户）里，坐标都一样。我们需要一种方式让每个词能"看看周围的词"来理解自己的含义。

TL;DR：Attention（注意力机制）让模型在处理每个词时能"关注"到句子中所有其他词，从而理解上下文。它是 Transformer 架构的核心，也是 LLM 能"理解"语言的根本原因。

故事：从"排队"到"并行"的革命

在 Transformer 之前（2017 年以前），处理语言的主流模型是 RNN（循环神经网络，Recurrent Neural Network）。RNN 的工作方式就像排队过安检：

  • 每个词必须一个一个通过模型
  • 第 100 个词必须等前面 99 个词都处理完才能开始
  • 而且到第 100 个词的时候，第 1 个词的信息已经衰减得很弱了（梯度消失，Vanishing Gradient）

这就像传话游戏——100 个人传一句话，到最后一个人时内容已经面目全非。

2017 年，Google Brain 团队发表了一篇改变一切的论文：《Attention is All You Need》（Vaswani et al.）。

他们提出的 Transformer 架构用一种全新的方式处理语言——不再排队，而是同时开多个安检通道。每个词可以直接"看到"句子中的所有其他词，不需要等前面的词处理完。

图 0C-1：RNN（排队） vs Transformer（并行）

Attention 的直觉：教室里的学生

想象一个教室里有一排学生，每人手里举着一个词牌：

「 他 走进 了 那家 银行 存了 一笔 钱 」

当模型处理"银行"这个词时，它需要理解这里的"银行"是金融机构还是河岸。怎么判断？看周围的词。"存了"、"钱"这些词强烈暗示这是金融机构。Attention 就是这个"看周围"的机制——它让每个词都能查看句子中的所有其他词，并决定应该特别关注哪些词。

Q / K / V 三兄弟——用图书馆来理解

Attention 的核心是三个向量：Query（查询）、Key（键）、Value（值）。这三个概念可以用图书馆来理解：

概念图书馆类比实际含义
Query (Q)你走进图书馆，说"我要找关于 AI Agent 的书"当前词在"寻找"什么信息
Key (K)每本书封面上的标签/关键词每个词"提供"什么信息
Value (V)每本书的实际内容每个词的实际语义信息

Attention 的计算过程：

  1. 拿你的搜索需求（Q）和每本书的标签（K）对比 → 得到"相关度分数"
  2. 用 softmax 把分数变成概率 → 加起来 = 100%
  3. 按概率从每本书（V）中提取内容 → 最相关的书贡献最多
// Attention 的数学（不用怕，就是三步）：

Step 1: 相关度 = Q × K^T / √d    ← Q 和每个 K 做点积，除以缩放因子
Step 2: 权重 = softmax(相关度)     ← 变成概率分布 [0.05, 0.60, 0.30, 0.05]
Step 3: 输出 = 权重 × V            ← 按权重加权求和所有 V

// 一行公式：
Attention(Q, K, V) = softmax(QK^T / √d_k) × V

Multi-Head Attention：不只一个人搜索

一个人搜图书馆可能只关注一个角度。但如果同时派 8 个人去搜——一个关注语法关系，一个关注语义相似性，一个关注指代关系，一个关注情感色彩……最后把 8 个人的搜索结果合并起来，理解就更全面了。这就是 Multi-Head Attention（多头注意力）。

Transformer 内部其实就是把以上的 Attention 操作堆叠很多层（GPT-3 有 96 层），每层都做：

  1. Multi-Head Self-Attention — 每个词看看其他所有词
  2. Feed-Forward Network — 对每个词的信息做进一步加工
  3. Layer Normalization + Residual Connection — 稳定训练过程

💡 为什么叫 "Attention is All You Need"？
论文标题的意思是：以前大家以为处理语言必须用循环（RNN）或卷积（CNN），但实际上只靠 Attention 就够了。Transformer 完全抛弃了循环和卷积结构，仅用 Attention 就达到了更好的效果。这在当时是一个非常大胆的主张，但事实证明他们是对的——现在几乎所有主流 LLM 都基于 Transformer。
⚠️ 常见误区
"Attention 就是让模型'注意'重要的词" — 这个说法太模糊了。精确地说，Attention 是一个加权信息聚合机制：通过 Q·K 计算相关度，softmax 归一化成权重，再用权重对 V 做加权求和。记住三个具体步骤，而不是一个模糊的"注意"概念。

位置编码（Position Encoding）——让 Transformer 知道词序

你可能注意到了一个问题：Transformer 用 Attention 让所有词同时互相关注——但它怎么知道词的顺序呢？

想想看："我爱你"和"你爱我"，如果只看词的集合 {我, 爱, 你}，它们完全一样！但意思完全不同。没有位置信息，Transformer 就是个"词袋模型"——它不知道谁在前谁在后。

类比：剧场的座位号。没有座位号，观众只是一群人；有了座位号，每个人都有确定的位置。位置编码就是给每个 token 加上它的"座位号"。

  没有位置编码：                      有位置编码：

  [我] [爱] [你]                     [我+pos₁] [爱+pos₂] [你+pos₃]
  [你] [爱] [我]                     [你+pos₁] [爱+pos₂] [我+pos₃]
       ↓                                    ↓
  Transformer 看起来一样！            Transformer 能区分了！

具体做法：在 token 经过 Embedding 变成向量后，加上一个等长的位置向量，再送入 Transformer。

位置编码的演进

原始 Transformer（2017）使用正弦余弦编码（Sinusoidal Encoding）——用固定的数学公式生成位置向量。简单但有效，不需要学习。

现代 LLM 用更先进的方案：

  • RoPE（Rotary Position Embedding，旋转位置编码）——LLaMA、Qwen、Mistral 等主流开源模型都在用。核心思想：通过旋转向量来编码相对位置。两个词的相对距离直接体现在它们向量的旋转角度差上。好处：天然支持外推到更长的序列。
  • ALiBi（Attention with Linear Biases）——不修改 Embedding，而是直接在 Attention 分数上加一个距离惩罚：两个词距离越远，Attention 分数越低。简单粗暴但有效。
💡 为什么位置编码很重要
位置编码直接影响模型能处理多长的上下文。当模型需要处理超出训练长度的文本时（比如 4K 训练但推理时用 128K），位置编码的外推能力就是关键瓶颈。后面 Chapter 12 讲 "Lost in the Middle" 现象时会再回来讨论。

Transformer 的三种变体——Encoder、Decoder 和它们的组合

原始的 Transformer 论文（"Attention Is All You Need"）其实包含两个部分：Encoder（编码器）负责"读懂"输入，Decoder（解码器）负责"生成"输出。但后来的研究发现，你不一定两个都要！

三种架构

① Encoder-only（BERT 系列）—— 阅读理解考试

只用 Encoder。特点：每个词能同时看到前后所有词（双向注意力）。训练方式："完形填空"——随机遮住一些词，让模型猜。

擅长：理解文本。用于文本分类、情感分析、命名实体识别（NER）、语义搜索。

类比：做阅读理解题——先看完全文，再回答问题。

② Decoder-only（GPT 系列）—— 即兴演讲

只用 Decoder。特点：每个词只能看到前面的词（因果注意力 / 自回归）。训练方式："接龙"——给前面的词，预测下一个词。

擅长：生成文本。这就是当今所有主流 LLM 的架构——GPT-4、Claude、LLaMA、Qwen、Gemini 全部是 Decoder-only。

类比：即兴演讲——你只能基于已经说过的话继续说，不能偷看后面的稿子。

③ Encoder-Decoder（T5、BART）—— 同声翻译

两个都用。Encoder 读完整个输入，Decoder 基于 Encoder 的理解逐步生成输出。

擅长：输入和输出格式不同的任务——翻译、摘要、问答。

类比：翻译——先读完整段原文（Encoder），再逐句翻译出来（Decoder）。

架构代表模型注意力方向训练方式擅长
Encoder-onlyBERT, RoBERTa双向（看所有词）遮词填空 (MLM)理解、分类、搜索
Decoder-onlyGPT, Claude, LLaMA单向（只看前文）预测下一个词 (CLM)文本生成
Encoder-DecoderT5, BART, mBARTEncoder 双向 + Decoder 单向序列到序列翻译、摘要
💡 关键洞察
GPT、Claude 都是 Decoder-only 架构——这就是为什么它们以"预测下一个 token"的方式工作，也只能看到当前位置之前的上下文。下一章 0D 讲 LLM 训练时，"预测下一个 token"这个训练目标就是从 Decoder-only 架构直接决定的。理解了架构，训练方式就不奇怪了。

0D · LLM 的诞生——从 Transformer 到 ChatGPT

上一章我们理解了 Transformer——一种能同时处理所有词、让每个词"看到"所有其他词的架构。现在一个自然的问题出现了：如果把这个架构做到非常非常大，会发生什么？

TL;DR：LLM（大语言模型，Large Language Model）就是一个巨大的 Transformer。它通过三个阶段训练而成：①Pre-training 在海量文本上学语言模式，②SFT 用示范对话教它如何有用地回答，③RLHF 用人类偏好让回答更好。这三步合在一起，就把一个"自动补全器"变成了 ChatGPT。

故事：从自动补全到智能对话

你手机的输入法有"预测下一个词"的功能——你打"今天天气"，它会建议"很好"或"不错"。LLM 本质上就是一个极其高级的"预测下一个词"系统。

区别在于规模：

手机输入法GPT-4
参数量~几百万~1.8 万亿（估计）
训练数据几 GB 文本~13 万亿 tokens
能力简单的词语接龙推理、翻译、写代码、数学...

当"预测下一个词"的能力强到一定程度时，涌现（Emergence）发生了——模型突然展现出训练时没有明确教过的能力，比如逻辑推理、多步数学、代码理解。这就像一个读了所有书的人，不仅能背书，还能举一反三。

三阶段训练流水线

图 0D-1：LLM 的三阶段训练流程

① Pre-training（预训练）——读遍互联网

把几乎整个互联网的文本（书、维基百科、新闻、代码、论坛…）喂给模型。训练目标极其简单：给你一段话的前半句，预测下一个 token。比如给 "今天天气" 预测 "很好"。就这一个目标，通过万亿级 token 的训练，模型学到了语法、常识、推理、甚至世界知识。

类比：这就像让一个人读完全世界所有的书。读完之后他什么都知道一点，但你问他问题时，他可能会说一段维基百科式的干巴巴文字，而不是友好地和你对话。

② SFT（有监督微调，Supervised Fine-Tuning）——岗前培训

人工编写几万条"用户问→标准回答"的示范对话，教模型怎么好好回答。

// SFT 训练数据示例：
User: "帮我解释什么是量子计算"
Assistant: "量子计算是一种利用量子力学原理来处理信息的计算方式。
与传统计算机使用比特（0或1）不同，量子计算机使用量子比特
（qubit），它可以同时处于0和1的叠加态..."

类比：博览群书的学者完成了岗前培训——他现在知道面对用户提问时应该友好、准确、有结构地回答。

③ RLHF（人类反馈强化学习，Reinforcement Learning from Human Feedback）——用户打分优化

让模型对同一个问题生成多个回答，然后让人类标注员选"哪个更好"。关键的创新在于：人类不需要给每个回答打具体分数，只需要做比较判断（A比B好？还是B比A好？）。

这些比较数据被用来训练一个"奖励模型（Reward Model）"——它学会了自动给回答打分。然后用强化学习让 LLM 的回答朝着高分方向优化。

类比：餐厅大厨根据食客反馈不断调整菜品——"这道菜咸了点"、"那道菜的酱汁很棒"——慢慢地菜品越来越好吃。

🔑 理解 RLHF 需要的最小 RL 知识

RLHF 里的 "RL" 是强化学习（Reinforcement Learning）。一句话解释：一个智能体（Agent）通过试错来学习——做动作、拿反馈、调策略。

RL 的核心循环

          ┌─────────────────────────────────────┐
          │         强化学习循环                   │
          │                                     │
          │   状态 (State)                       │
          │       ↓                             │
          │   智能体选择 动作 (Action)              │
          │       ↓                             │
          │   环境给出 奖励 (Reward)               │
          │       ↓                             │
          │   更新 策略 (Policy)                  │
          │       ↓                             │
          │   进入新状态，重复...                   │
          └─────────────────────────────────────┘

类比：训狗。狗做了一个动作（action），你说"好孩子！"或者"不行！"（reward），狗调整行为（policy update）。你不需要给狗看说明书——它从反馈中学习。

映射到 LLM

在 RLHF 中，RL 的每个概念都有对应：

RL 概念LLM 中的对应具体是什么
状态 (State)上下文用户的 prompt + 已生成的 token
动作 (Action)生成 token从词表中选择下一个 token
奖励 (Reward)Reward Model 打分对整段回复的质量评分

PPO（Proximal Policy Optimization）——小步快跑

PPO 是 RLHF 中最常用的 RL 算法。你不需要理解它的数学——只需知道核心思想：每次更新都不要太激进。

类比：调整菜谱。如果你一次改了盐、糖、酱油、醋四种调料，菜变难吃了你都不知道是哪个改坏的。PPO 的做法是每次只微调一点，确保新的策略不会偏离旧策略太远。这样既能稳步改进，又不会"忘掉"之前学到的东西。

💡 连接后续章节
后面 Chapter 10 会讲 DPO 和 GRPO，它们是 PPO 的简化替代方案——不需要单独训练 Reward Model，直接从人类偏好数据学习。到时候你已经有 RL 基础了，会容易理解得多。
💡 关键洞察
Pre-training 决定了模型"知道什么"（知识上限），SFT 教它"怎么说"（输出格式），RLHF 让它"说得好"（输出质量）。三步缺一不可——只有 pre-training 的模型像一个博览群书但不会说人话的怪人；加了 SFT 后能正常对话但可能不够好；加了 RLHF 后才变成你见到的 ChatGPT。

01 · LLM 是怎么变成 Agent 的

经过 Part 0 的旅程，你已经知道了：神经网络是怎么学习的，文字是怎么变成数字的，Attention 是怎么让模型理解上下文的，以及 LLM 是怎么通过三阶段训练变成 ChatGPT 的。现在，让我们进入最激动人心的部分——把这个聪明的 LLM 变成一个真正能做事的 Agent。

直觉：从计算器到员工

Q: Agent 和 Workflow 的区别是什么？

标准答案：Workflow 是预定义的确定性流程（if-else 分支），所有路径在设计时就固定了。Agent 是非确定性的，它根据实时观察结果动态决策下一步。Workflow 适合流程稳定的场景，Agent 适合需要灵活应对的场景。

加分回答：实际系统中两者常结合——用 Workflow 做确定性骨架，在不确定的节点嵌入 Agent 做决策。比如退款流程用 Workflow，但"判断退款是否合理"这一步用 Agent。

常见踩坑：❌ "Agent 可以完全替代 Workflow"——不是。对于规则明确的流程，Workflow 更可靠、更便宜、更可审计。

Q: 模型和 Agent 的区别是什么？

标准答案：模型（LLM）是 Agent 的"大脑"——一个输入文本、输出文本的数学函数。Agent 是包含模型在内的完整系统，还包括工具调用、记忆管理、任务规划和执行循环。模型本身不能操作外部世界，Agent 可以。

02 · ReAct 循环：思考-行动-观察

TL;DR：ReAct = Reasoning + Acting。Agent 的每一步都是"先想再做，看结果，再想"——不断循环直到任务完成。这是 Agent 的心跳。

直觉：你去陌生城市找餐厅

🧠 思考：我饿了，想吃火锅，先查一下附近有什么
🔧 行动：打开大众点评搜索"附近火锅"
👁 观察：海底捞 4.8分，排队 2小时

🧠 思考：太久了，看第二家
🔧 行动：点开第二家
👁 观察：评分 4.6，不用排队，步行 10 分钟

🧠 思考：可以，走吧
🔧 行动：导航过去
✅ 完成：开吃！

你没有提前规划好所有步骤——你是边做边想，根据每步结果决定下一步。这就是 ReAct。

ReAct 循环

三种方法对比

方法过程缺点
Chain-of-Thought
只想不做		LLM 纯推理，不调用工具可能在"瞎猜"，没有真实数据
纯 Tool Calling
只做不想		收到请求就调工具，不推理不会反思结果，不会换方案
ReAct ✅
边想边做		每步先推理→再决定行动→看结果→再推理可能比较慢（多轮 LLM 调用）

ReAct 的变体模式

面试会问"ReAct 那几个模式有什么差别"。主要有三种：

1. 标准 ReAct
   思考 → 行动 → 观察 → 思考 → ...
   每一步都"想"，适合需要深度推理的任务

2. Act-only（纯行动）
   行动 → 观察 → 行动 → 观察 → ...
   省略思考步骤，靠工具结果直接驱动
   快但容易跑偏，适合简单任务

3. Plan-then-Execute
   规划 → [行动 → 观察] × N
   先做完整计划，再按计划执行
   适合步骤明确的任务，但不够灵活

对比：
标准 ReAct  = 边走边看路的司机（灵活但可能绕路）
Act-only   = 闷头开的司机（快但可能撞墙）
Plan-first = 先看完地图再出发（稳但不能应对堵车）

记忆的更新与遗忘

// 面试高频：传统记忆如何更新和遗忘？

更新策略：
① Append     — 新信息追加（最简单）
② Overwrite  — 新信息覆盖旧信息（如用户改了手机号）

LRU 原理与实现

// LRU = Least Recently Used（最近最少使用）
// 用于限制记忆容量：满了就淘汰最久没用的

class LRUCache {
  constructor(capacity) {
    this.capacity = capacity;
    this.map = new Map();  // Map 保持插入顺序
  }

  get(key) {
    if (!this.map.has(key)) return undefined;
    const value = this.map.get(key);
    this.map.delete(key);     // 删除
    this.map.set(key, value);  // 重新插入（变成最新）
    return value;
  }

  put(key, value) {
    if (this.map.has(key)) this.map.delete(key);
    this.map.set(key, value);
    if (this.map.size > this.capacity) {
      // 淘汰最老的（Map 的第一个）
      const oldest = this.map.keys().next().value;
      this.map.delete(oldest);
    }
  }
}

Late Chunking先编码整篇文档再切分保留全局上下文需要特定模型(Jina)长文档+Jina
💡 实际选型建议
入门/原型：递归分割（LangChain RecursiveCharacterTextSplitter），chunk_size=512, overlap=50
生产系统：父子索引 + Rerank。小块（128-256 token）做检索，返回父块（512-1024 token）给 LLM
如果已有 Rerank + Hybrid Search：语义分割的边际收益较小，递归分割就够了
记住：chunking 策略的影响 > embedding 模型的选择（研究表明）

向量数据库：Embedding 的家

上面我们把文本变成了向量（Embedding），也知道了怎么计算相似度。但实际场景中，你可能有几百万甚至上亿个文档向量。每来一个查询，都和所有向量逐一比较（暴力搜索）？一个 1536 维的向量比较一次就要 1536 次乘法——乘以一百万个文档，每次查询要十几亿次运算。太慢了！

类比：在图书馆找书。如果书随便堆在地上（暴力搜索），你得翻遍每一本。但如果书按主题分类、放在标了标签的书架上（索引），你先走到对的区域，再扫一遍那个书架就行了。向量数据库就是帮你"整理书架"的系统。

为什么不用传统数据库？

MySQL、PostgreSQL 这类关系型数据库擅长精确匹配：WHERE id = 123、WHERE name = '张三'。它们的索引（B-Tree）是为"等于""大于""小于"这类操作设计的。

但向量检索需要的是"在 1536 维空间中，找到离这个点最近的 10 个点"——这是一个完全不同的数学问题。传统索引结构在高维空间中效率极差（这叫"维度诅咒"）。所以需要专门的向量数据库。

ANN（Approximate Nearest Neighbor，近似最近邻）

核心思路：用一点点精度换巨大的速度提升。不追求找到"最最最近"的向量，而是找到"非常非常近"的向量。实践中，ANN 通常能达到 99%+ 的召回率，但速度比暴力搜索快 100 倍以上。

HNSW（Hierarchical Navigable Small World）——最流行的 ANN 算法

类比："六度分隔"理论。在社交网络里，任何两个人之间平均只隔 6 个人。HNSW 就是给向量建一个类似的"社交网络"——每个向量和一些"邻居"建立连接，搜索时沿着连接跳转，很快就能找到目标附近。

  HNSW 多层图结构：

  Layer 2 (稀疏)    A ──────────────── D        ← 高速公路：少数节点，远距离跳转
                    │                  │
  Layer 1 (中等)    A ─── B ────── D ── E       ← 主干道：更多节点
                    │     │        │    │
  Layer 0 (稠密)    A ─ B ─ C ─ D ─ E ─ F ─ G   ← 本地街道：所有节点，短距离连接

  搜索过程：从顶层开始 → 快速定位大致区域 → 逐层下沉 → 在底层精确搜索

Hybrid Search（混合检索）

// 问题：向量检索擅长语义匹配，但对精确关键词匹配弱
// BM25 擅长关键词匹配，但不懂语义

用户问："OpenClaw 的 Gateway 端口是多少？"
  向量检索 → 可能返回"OpenClaw 架构介绍"（语义相关但没答案）
  BM25    → 可能返回"端口配置：18789"（精确匹配）

// Hybrid Search = 向量 + BM25 的融合
final_score = α × vector_score + (1-α) × bm25_score
  // α 通常取 0.5~0.7
  // 用 RRF (Reciprocal Rank Fusion) 融合两个排名列表效果更好

Rerank（重排序）

// 为什么需要 Rerank？
// 初始检索（embedding 向量匹配）是"粗筛"——快但不够精
// Rerank 是"精排"——慢但更准

初始检索（embedding）→ Top 20 候选
        ↓
Rerank 模型（cross-encoder）→ 重新打分排序
        ↓
取 Top 3-5 → 送给 LLM 生成

// 为什么 Rerank 比 embedding 更准？
Embedding（Bi-Encoder）：  问题和文档分别编码，用余弦比较
  快（可以预计算），但精度有限

Rerank（Cross-Encoder）：  问题+文档拼在一起，同时编码
  慢（不能预计算），但精度更高
  因为它能看到问题和文档之间的细粒度交互

// 常用 Rerank 模型：
// Cohere Rerank, bge-reranker-v2, Jina Reranker

指标评估什么怎么算

🎯 面试高频题
Q: 一个标准的 RAG 系统包含哪些模块？完整流程是什么？

标准答案：分两个阶段——
离线阶段：①文档加载（PDF/网页/数据库）→ ②文档切分（Chunking，递归分割或语义分割）→ ③文本向量化（Embedding，如 text-embedding-3-small）→ ④向量索引（存入向量数据库如 Milvus/Pinecone/Chroma）
在线阶段：①用户提问 → ②问题向量化 → ③向量检索（Top-K 相似块）→ ④Rerank 重排序 → ⑤拼接 Prompt（参考文档+问题）→ ⑥LLM 生成带引用的回答

加分回答：提到 Hybrid Search（向量+BM25）、父子索引、metadata filtering（按时间/类型过滤）、评估指标（Recall@K, Faithfulness, Answer Relevance）。

常见踩坑：❌ 只说"检索+生成"两步——缺少 chunking、embedding、rerank 等关键环节；❌ 分不清 Bi-Encoder（embedding检索）和 Cross-Encoder（rerank）的区别。
🎯 面试高频题
Q: Chunk size 怎么选？为什么很重要？

标准答案：Chunk size 直接影响检索质量。太小（如 100 token）：检索精准但缺上下文，LLM 没有足够信息生成好的回答。太大（如 2000 token）：上下文完整但检索不精准，且占用更多 context window。通常 256-512 token 是好的起点，需要根据具体数据和任务调优。

加分回答：提到父子索引（小块检索、返回大块），以及 overlap（chunk 之间有 10-20% 的重叠，避免关键信息被切断）。

常见踩坑：❌ 给一个固定数值不解释——chunk size 没有万能值，取决于文档类型和任务。
🎯 面试高频题
Q: Claude Code 的记忆架构是什么？为什么不用 RAG 检索代码而是用 grep？

标准答案：Claude Code 的记忆分三层：①Context Window（当前对话）②CLAUDE.md 文件（项目级指令）③Memory 文件（持久化笔记）。它用 grep/ripgrep 而非 RAG 搜索代码，因为：代码搜索通常需要精确匹配（函数名、变量名），grep 在这方面比向量检索更准更快。向量检索擅长语义匹配（"类似的意思"），但代码中需要的是"精确的符号匹配"。

加分回答：补充说"上下文≠记忆"——context window 是临时的，每次对话结束就没了；真正的记忆需要持久化到文件系统（OpenClaw 用 MEMORY.md + memory/*.md）。

常见踩坑：❌ "RAG 比 grep 更高级所以应该用 RAG"——选工具要看场景，精确匹配场景 grep 更优。

🔥 幻觉（Hallucination）详解

// 幻觉 = LLM 生成了看起来正确但实际错误的内容
// 研究表明：领先 AI 模型在科学摘要中幻觉率高达 31%

两大类型（最新分类法）：
① 知识型幻觉（Knowledge-based）
   → 事实错误："爱因斯坦于 1950 年获诺贝尔奖"（错，是 1921）
   → 引用虚构不存在的论文/链接
   → 原因：训练数据缺失或矛盾

② 逻辑型幻觉（Logic-based）
   → 推理过程看起来对但结论错
   → 多步推理中错误累积
   → 原因：自回归模型的"模式延续"倾向

根本成因（从架构角度）：
① 自回归生成 — 模型预测下一个 token 时倾向于"自信补全"
② 知识缺口 — 对长尾知识或最新信息覆盖不足
③ 模式驱动 — 生成了语法正确但事实错误的续写
④ Context 稀释 — 长上下文中 Attention 衰减（Lost in the Middle）
⑤ 训练数据噪声 — 互联网数据本身包含错误信息

7 层缓解策略（从轻到重）：
① Prompt 工程     — 明确指令："只基于给定信息回答，不确定则说不知道"
② RAG             — 检索真实文档作为上下文（最有效的单一策略）
③ Grounding       — 要求模型引用来源，每句话标注出处
④ Self-check      — 生成后让模型自己验证（"请检查上述回答是否有事实错误"）
⑤ Constrained 输出 — 结构化输出 + 限制回答范围
⑥ 外部验证       — 自动化 fact-checker 验证关键事实
⑦ Fine-tuning    — 用领域专家标注的数据微调模型

// 实际生产中通常组合使用：RAG + Grounding + Self-check

from openai import OpenAI
import numpy as np

client = OpenAI()

# ── 1. 准备文档并切分 ──
documents = [
    "OpenClaw 的 Gateway 默认端口是 18789",
    "OpenClaw 用 WebSocket 协议通信，不是 HTTP",
    "Agent 的 System Prompt 包含 Tools、Skills、Bootstrap 文件",
    "Session 会在每天凌晨 4 点自动重置",
]

# ── 2. 文本 → 向量 ──
def embed(texts):
    resp = client.embeddings.create(model="text-embedding-3-small", input=texts)
    return [r.embedding for r in resp.data]

doc_vectors = embed(documents)

# ── 3. 检索 ──
def search(query, top_k=2):
    q_vec = embed([query])[0]
    scores = [np.dot(q_vec, dv) for dv in doc_vectors]
    top_idx = np.argsort(scores)[-top_k:][::-1]
    return [documents[i] for i in top_idx]

# ── 4. RAG 生成 ──
question = "OpenClaw 用什么协议？"
context = search(question)

answer = client.chat.completions.create(
    model="gpt-4",
    messages=[
        {"role": "system", "content": f"基于以下参考文档回答：\n{chr(10).join(context)}"},
        {"role": "user", "content": question}
    ]
)
print(answer.choices[0].message.content)
# → "OpenClaw 使用 WebSocket 协议通信"

  DeepSeek 提出，PPO 的简化版
  不需要 Reward Model，用组内样本相对排名做优化
  比 DPO 更灵活，比 PPO 更简单

KV Cache（键值缓存）——为什么长对话吃显存？

先看一个问题：当模型生成第 100 个 token 时，它需要对前 99 个 token 都算一遍 Attention。生成第 101 个 token 时，又要对前 100 个 token 算一遍。这里有大量重复计算——前 99 个 token 的 K 和 V 向量，在生成第 100 和第 101 个 token 时完全一样，为什么要算两次？

// 没有 KV Cache：每生成一个新 token，重新计算所有 K/V

生成 token 1:  计算 K₁, V₁
生成 token 2:  计算 K₁, V₁, K₂, V₂           ← K₁V₁ 重复算了！
生成 token 3:  计算 K₁, V₁, K₂, V₂, K₃, V₃   ← K₁V₁, K₂V₂ 重复算了！
...
生成 token N:  计算 K₁..Kₙ, V₁..Vₙ            ← O(N²) 的重复计算

// 有 KV Cache：只算新 token 的 K/V，旧的从缓存读

生成 token 1:  计算 K₁, V₁ → 存入缓存
生成 token 2:  读缓存 K₁, V₁ + 计算 K₂, V₂ → 存入缓存
生成 token 3:  读缓存 K₁..₂, V₁..₂ + 计算 K₃, V₃ → 存入缓存
...
生成 token N:  读缓存 K₁..ₙ₋₁, V₁..ₙ₋₁ + 计算 Kₙ, Vₙ  ← O(N) 新计算

类比：想象一个法庭书记员。没有 KV Cache 就像每次有人发言，书记员都要从头把整个庭审记录重新读一遍再记新内容。有了 KV Cache，书记员保持着自己的笔记，每次只需要在笔记末尾追加新内容。

⚠️ KV Cache 的代价：显存
KV Cache 用空间换时间。每个 token 需要缓存所有层、所有 head 的 K 和 V 向量。

粗算：一个 7B 模型，32 层 × 32 heads × 128 维 × 2（K+V）× 2 字节（FP16）= 每个 token 约 0.5 MB
一个 128K context window 的对话 → KV Cache 约 64 GB 显存！

这就是为什么你和 AI 聊天聊久了会看到"上下文已满"——很多时候不是模型"忘了"，而是 KV Cache 的显存用完了。

Context Window 管理策略

Context Window（上下文窗口）是模型一次能处理的最大 token 数。GPT-4o 是 128K，Claude 是 200K。听起来很大，但实际上要精打细算：

// Context Window 的预算分配（Budget Allocation）

┌──────────────────────────────────────────────────┐
│                 128K Context Window               │
├────────────┬───────────┬──────────────┬───────────┤
│ System     │ Tools /   │ Conversation │ Current   │
│ Prompt     │ Skills    │ History      │ Query     │

│ (25K)      │ (19K)     │ (64K)        │ (19K)     │

│ System Prompt: 角色定义、规则、SOUL.md 等          │
│ Tools/Skills: 工具定义、技能文件内容              │
│ History: 之前的对话消息                          │
│ Current: 用户当前的问题 + 本轮生成的回答          │
└──────────────────────────────────────────────────┘

策略做法优点缺点适用场景
截断
(Truncation)		 丢弃最旧的消息 实现简单，零延迟 可能丢失关键上下文 闲聊、低风险对话
摘要
(Summarization)		 用 LLM 把旧对话压缩成摘要 保留关键信息，压缩率高 摘要可能遗漏细节，有额外 API 调用成本 长期项目、多轮任务
滑动窗口
(Sliding Window)		 只保留最近 N 条消息 行为可预测，实现简单 N 之前的信息全部丢失 客服对话、短任务
选择性保留
(Selective Retention)		 保留重要消息（工具结果、关键决策），丢弃闲聊 在有限空间内最大化信息密度 需要判断"什么是重要的" Agent 任务、编程辅助 
// OpenClaw 的 Context Engine 工作原理（简化版）

function assemble(msgs, budget) {
  const system = msgs.filter(m => m.role === 'system')  // 永远保留
  const recent = msgs.slice(-10)                        // 最近 10 条必须保留
  let remaining = budget - tokenCount(system) - tokenCount(recent)

  // 中间的消息按重要性排序
  const middle = msgs.slice(system.length, -10)
    .map(m => ({ ...m, score: importance(m) }))
    .sort((a, b) => b.score - a.score)

  // 贪心装入：优先保留重要的消息
  const selected = []
  for (const m of middle) {
    if (tokenCount(m) <= remaining) {
      selected.push(m)
      remaining -= tokenCount(m)
    }
  }

  // 按原始顺序重新排列，保持对话连贯性
  return [...system, ...selected.sort(byIndex), ...recent]
}

// importance() 的评分规则：
//   tool_result（工具调用结果）→ 高分（包含事实数据）
//   user 消息带代码/文件 → 高分（可能是关键上下文）
//   assistant 的短回答 → 低分（"好的"、"明白了"）
//   user 的闲聊 → 低分（"谢谢"、"嗯"）

Lost in the Middle（中间丢失效应）

2023 年，斯坦福的 Liu et al. 发表了一篇重要论文 "Lost in the Middle: How Language Models Use Long Contexts"，揭示了一个关键现象：模型对上下文中不同位置的信息，关注度差异巨大。

// Lost in the Middle：注意力权重在不同位置的分布

位置:  [开头] ────────── [中间] ────────── [结尾]
权重:  ████              ██                ████████

开头（system prompt）→ 权重高（位置编码优势）
中间（旧的对话历史）→ 权重低（最容易被"遗忘"）
结尾（最近的消息）  → 权重高（近因效应）

// 这就是为什么多轮对话越长，Agent 越容易"忘记"中间说过的话
💡 工程实践建议
1. 重要信息放两头：System Prompt（开头）放角色定义和核心规则，最近的消息（结尾）放当前任务上下文。不要把关键指令埋在长对话的中间。

2. RAG 检索结果要排序：如果你检索了 5 篇相关文档，把最相关的放在第一个，而不是随机排列。模型会更关注第一篇和最后一篇，中间的容易被忽略。这就是为什么 RAG 需要 Reranker（重排序器）。

3. Compaction（压缩）优于 Truncation（截断）：与其直接丢弃中间的消息，不如把它们压缩成摘要放在开头位置。OpenClaw 的做法是：当 context 使用超过 80%，自动触发 compaction，把旧对话压缩为一段总结。
🎯 面试高频题
Q: Attention 本质是什么？从向量空间变换角度解释。

标准答案：Attention 本质是一个加权信息聚合机制。每个 token 通过 Q/K/V 三个线性变换映射到不同的向量空间：Q（查询）代表"我在找什么"，K（键）代表"我有什么"，V（值）代表"我的内容"。通过 Q·K^T 计算相关度，softmax 归一化后加权求和 V，得到融合了全局信息的表示。数学上：Attention(Q,K,V) = softmax(QK^T/√d_k)V。

加分回答：提到 Multi-Head Attention（多个 head 关注不同方面，如语法/语义/指代），√d_k 缩放防止梯度消失，以及 KV Cache 如何优化推理速度。

常见踩坑：❌ 只说"Attention 让模型关注重要的部分"——太模糊，要说出 Q/K/V 的计算过程和 KV Cache 的工程意义。

Q: 多轮对话的存储、记忆解决方案？

标准答案：短期记忆存在 context window 里（messages 数组），长期记忆持久化到文件/数据库。具体实现：①JSONL 文件存储完整对话历史②context window 满时用 compaction 压缩旧消息为摘要③重要信息主动写入 Memory 文件。OpenClaw 的做法：session transcript 用 JSONL，长期记忆用 MEMORY.md + memory/YYYY-MM-DD.md。

18 · Skills 系统

TL;DR：Skill = Agent 的专业手册，按需加载不占 context。三级优先：Workspace > Managed > Bundled。

渐进式披露（Progressive Disclosure）

// 问题：Agent 有 50 个 Skill，每个 SKILL.md 几千字
// 全放进 context window → 几万 token → 爆了

// 解决：渐进式披露
Level 1: System Prompt 只包含目录
  "可用技能：weather(天气), github(GitHub)..."
  → 每个 Skill 只占 ~50 tokens

Level 2: 需要时读取 SKILL.md
  Agent → read("~/.openclaw/skills/weather/SKILL.md")
  → 只有用到时才加载完整指令

// 类似 MCP 的 Progressive Discovery（2026.01 新增）
// Skill 更早实现了这种"按需加载"的设计
🎯 面试高频题
Q: Skill 了解吗？渐进式披露是怎么实现的？Skill 里面有没有工具？

标准答案：Skill 是 Agent 的行为指令文件（SKILL.md），定义"遇到什么场景，怎么做"。渐进式披露：system prompt 只注入 Skill 目录（名称+描述+路径），Agent 需要时再用 read 工具加载完整内容。Skill 本身不是工具，但 Skill 的指令中通常会引导 Agent 调用特定工具（如 exec、web_search）。

加分回答：提到三级加载优先级（Workspace > Managed > Bundled）和 MCP 在 2026.01 也加入了类似的 Progressive Discovery（工具搜索），解决 context 污染问题。

常见踩坑：❌ "Skill 就是工具"——不是，Skill 是指令/方法论，工具是能力。
