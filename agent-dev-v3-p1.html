Part 1: Agent 原理
从 LLM 到 Agent 的核心概念 - 10 个关键知识点

01 · LLM 是怎么变成 Agent 的

经过 Part 0 的旅程,你已经知道了:神经网络是怎么学习的,文字是怎么变成数字的,Attention 是怎么让模型理解上下文的,以及 LLM 是怎么通过三阶段训练变成 ChatGPT 的。现在,让我们进入最激动人心的部分--把这个聪明的 LLM 变成一个真正能做事的 Agent。

TL;DR:Agent = LLM + Loop + Action(Tools) + Memory + Planning。LLM 是"大脑",其余四个组件(LAMP 🪔)让它从"只能说"变成"能做事"。

直觉:从计算器到员工

想象你有一个超级聪明的朋友--他读过所有的书,能回答任何问题。但有个问题:他被关在一个房间里,只能通过纸条和你交流。

// 纯 LLM(被关在房间里的天才)
你写纸条:  "帮我订明天去上海的机票"
他回纸条:  "好的,你可以去携程网搜索..."  ← 他只能"说",不能"做"

// Agent(放出来的天才,有手有脚有记忆)
你说:      "帮我订明天去上海的机票"
他:         打开携程 → 搜索航班 → 比价 → 下单 → 告诉你 "订好了,东航 MU5101,¥680"

这就是 LLM 和 Agent 的核心区别:LLM 是大脑,Agent 是一个完整的"人"。

Agent 的精确定义

AI Agent(AI 智能体)是一个能够自主感知环境、做出决策、并采取行动来实现目标的 AI 系统。

核心差异:前三者都需要人类预定义所有可能的路径,Agent 能自主决策面对未知情况。

Agent 的自主性(Autonomy)

Agent 的核心特征是目标驱动执行(Goal-driven Execution)--你给它一个目标,它自己决定怎么实现:

// 普通 LLM 调用:你决定每一步
response1 = llm.chat("搜索北京天气")     ← 你指挥
response2 = llm.chat("根据天气推荐穿搭") ← 你指挥
response3 = llm.chat("生成朋友圈文案")   ← 你指挥

// Agent:你只给目标,它自己决定怎么做
agent.run("帮我发一条关于今天天气的朋友圈")
  → Agent 自己决定:先查天气 → 再想穿搭 → 再写文案 → 发布
  → 如果天气 API 挂了:换一个 API 试试
  → 如果文案不够好:自己重写一遍

为什么 Agent 比 ChatBot 复杂得多?

ChatBot 的复杂度 = O(场景数量)
  // 每多一个场景,多写一组规则
  // 100 个场景 = 100 组规则

Agent 的复杂度 = O(能力组合 × 环境不确定性)
  // 10 个工具的组合方式 = 10! 种
  // 每步结果都可能偏离预期
  // 需要实时重新规划

这就是为什么 Agent 需要 LAMP 四个组件:
  Loop     → 持续运转,处理不确定性
  Tools    → 与外界交互的能力
  Memory   → 记住上下文和历史
  Planning → 拆解目标为可执行步骤

Q: 什么是 AI Agent?和普通 LLM API 调用有什么本质区别?

标准答案:AI Agent 是能自主感知环境、决策和行动的 AI 系统。核心组件是 LLM + Loop + Tools + Memory + Planning。与普通 LLM 调用的区别在于:LLM 调用是"一问一答",人驱动每一步;Agent 是"给目标自己跑",有自主循环、能调用工具、能记住上下文、能拆解任务。

🎯 面试高频题
Q: Agent 和 Workflow 的区别是什么?

标准答案:Workflow 是预定义的确定性流程(if-else 分支),所有路径在设计时就固定了。Agent 是非确定性的,它根据实时观察结果动态决策下一步。Workflow 适合流程稳定的场景,Agent 适合需要灵活应对的场景。

加分回答:实际系统中两者常结合--用 Workflow 做确定性骨架,在不确定的节点嵌入 Agent 做决策。比如退款流程用 Workflow,但"判断退款是否合理"这一步用 Agent。

常见踩坑:❌ "Agent 可以完全替代 Workflow"--不是。对于规则明确的流程,Workflow 更可靠、更便宜、更可审计。
🎯 面试高频题
Q: 模型和 Agent 的区别是什么?

标准答案:模型(LLM)是 Agent 的"大脑"--一个输入文本、输出文本的数学函数。Agent 是包含模型在内的完整系统,还包括工具调用、记忆管理、任务规划和执行循环。模型本身不能操作外部世界,Agent 可以。

加分回答:类比:模型 = CPU,Agent = 整台电脑(CPU + 硬盘 + 网卡 + 操作系统)。CPU 再强也不能单独上网,得有完整系统才行。

02 · ReAct 循环:思考-行动-观察

TL;DR:ReAct = Reasoning + Acting。Agent 的每一步都是"先想再做,看结果,再想"--不断循环直到任务完成。这是 Agent 的心跳。

直觉:你去陌生城市找餐厅

🧠 思考:我饿了,想吃火锅,先查一下附近有什么
🔧 行动:打开大众点评搜索"附近火锅"
👁 观察:海底捞 4.8分,排队 2小时

🧠 思考:太久了,看第二家
🔧 行动:点开第二家
👁 观察:评分 4.6,不用排队,步行 10 分钟

🧠 思考:可以,走吧
🔧 行动:导航过去
✅ 完成:开吃!

你没有提前规划好所有步骤--你是边做边想,根据每步结果决定下一步。这就是 ReAct。

ReAct 循环

三种方法对比

方法过程缺点
Chain-of-Thought
只想不做		LLM 纯推理,不调用工具可能在"瞎猜",没有真实数据
纯 Tool Calling
只做不想		收到请求就调工具,不推理不会反思结果,不会换方案
ReAct ✅
边想边做		每步先推理→再决定行动→看结果→再推理可能比较慢(多轮 LLM 调用)

ReAct 的变体模式

面试会问"ReAct 那几个模式有什么差别"。主要有三种:

1. 标准 ReAct
   思考 → 行动 → 观察 → 思考 → ...
   每一步都"想",适合需要深度推理的任务

2. Act-only(纯行动)
   行动 → 观察 → 行动 → 观察 → ...
   省略思考步骤,靠工具结果直接驱动
   快但容易跑偏,适合简单任务

3. Plan-then-Execute
   规划 → [行动 → 观察] × N
   先做完整计划,再按计划执行
   适合步骤明确的任务,但不够灵活

对比:
标准 ReAct  = 边走边看路的司机(灵活但可能绕路)
Act-only   = 闷头开的司机(快但可能撞墙)
Plan-first = 先看完地图再出发(稳但不能应对堵车)

如何避免无限循环?

这是面试高频问题。Agent 陷入死循环是生产中最常见的问题之一:

// 死循环示例
🧠 思考:搜索结果不对,换个关键词再搜
🔧 行动:web_search("xxx")
👁 观察:还是找不到
🧠 思考:再换一个关键词...
🔧 行动:web_search("yyy")
👁 观察:还是找不到
→ 无限循环...

// 四种防护策略
1. 最大轮次限制(Max Iterations)
   max_turns = 25  // 超过就强制停止

2. 超时机制(Timeout)
   timeout = 300  // 5 分钟超时

3. 重复检测(Loop Detection)
   如果连续 3 次调用同一个工具且参数相似 → 强制终止

4. Token 预算(Budget)
   max_tokens = 100000  // 花完就停

Checkpoint / Resume:中断恢复

// Agent 在执行长任务时,每完成一步保存状态
{
  task: "分析 10 个竞品",
  checkpoint: {
    completed: ["竞品A", "竞品B", "竞品C"],
    current: "竞品D",
    remaining: ["竞品E", ... "竞品J"],
    partial_results: { ... }
  }
}

// 如果中断(超时/崩溃),可以从 checkpoint 恢复
agent.resume(checkpoint)  → 从"竞品D"继续,不用从头来
🎯 面试高频题
Q: 什么是 ReAct?Agent 如何避免无限循环调用工具?

标准答案:ReAct = Reasoning + Acting,Agent 交替进行推理和行动:先思考下一步做什么→执行→观察结果→再思考。避免无限循环的策略:1设置最大轮次(如 25 轮)2设置超时(如 5 分钟)3重复调用检测(连续相同工具调用则终止)4Token 预算限制。

加分回答:提到 ReAct 的三种变体(标准/Act-only/Plan-then-Execute)的区别和适用场景。提到 OpenClaw 的 Agent Loop 默认超时是 48 小时,通过 per-session + global queue 序列化来保证安全。

常见踩坑:❌ "设一个 max_turns 就够了"--不够,需要多层防护;❌ 把 ReAct 和 React(前端框架)搞混。

TL;DR:LLM 不能执行工具--它只是"说"想调用什么。真正执行的是外部系统。Tool Calling 是基础协议,MCP 是工具连接标准,Skill 是行为指令标准,A2A 是 Agent 间通信标准,ACP 是 OpenClaw 的 Agent 间任务委派方案。

Function Calling 协议详解

当你调用 OpenAI/Anthropic 的 API 时,工具调用实际上是这样工作的:

// Step 1: 你告诉 LLM 有哪些工具可用
{
  "model": "gpt-4",
  "messages": [{ "role": "user", "content": "北京天气怎么样?" }],
  "tools": [{
    "type": "function",
    "function": {
      "name": "get_weather",
      "description": "获取指定城市的天气信息",
      "parameters": {
        "type": "object",
        "properties": {
          "city": { "type": "string", "description": "城市名" }
        }
      }
    }
  }]
}

// Step 2: LLM 返回"我想调用这个工具"(不是直接执行!)
{
  "role": "assistant",
  "tool_calls": [{
    "id": "call_abc123",
    "function": { "name": "get_weather", "arguments": "{\"city\":\"北京\"}" }
  }]
}

// Step 3: 你的代码执行工具,把结果送回去
{
  "role": "tool",
  "tool_call_id": "call_abc123",
  "content": "北京今天 22°C,晴,微风"
}

// Step 4: LLM 根据工具结果生成最终回答

ACP 是什么?

ACP（Agent Communication Protocol，Agent 通信协议）是 OpenClaw 的 Agent 间任务编排方案（平台特定实现，不是行业通用标准）,解决一个具体问题:一个 Agent 怎么把任务交给另一个 Agent 执行,然后拿回结果？

举个真实例子--你在飞书群里跟 Cove(OpenClaw 的主 Agent)说:

// 真实场景:Cove 通过 ACP 委派任务给 Claude Code

你:"帮我写一个 Python 爬虫,抓取 Hacker News 前 10 条新闻"

Cove(主 Agent):
  1 判断:这是一个编码任务,我自己不擅长,交给 Claude Code
  2 通过 ACP 发起委派:
     sessions_spawn({
       runtime: "acp",
       agentId: "claude-code",
       task: "写一个 Python 爬虫抓取 HN 前 10 条...",
       mode: "run"   // 一次性任务
     })

Claude Code(ACP Agent):
  1 在自己的沙箱工作空间里写代码
  2 运行、调试、测试
  3 完成后通过 ACP 返回结果

Cove:
  1 收到 Claude Code 的完成通知
  2 把结果整理后回复给你
💡 ACP vs A2A 的区别
A2A(Agent-to-Agent Protocol)是 Google 提出的通用 Agent 间通信标准,强调的是 Agent 的发现(Agent Card)和对等通信--两个 Agent 地位平等,互相发消息。

ACP(Agent Communication Protocol)更侧重任务委派关系--有明确的父子关系(Parent Agent 委派任务,Child Agent 执行并回报)。它更像一个"工头-工人"模型,适合编排(orchestration)场景。

实际中两者可以共存:A2A 负责跨平台 Agent 发现,ACP 负责同一平台内的任务编排。

Skill 和 MCP 的区别

// MCP 给 Agent 的是"能力"(access)
MCP: "你可以调用 get_weather、send_email、query_db..."

// Skill 给 Agent 的是"纪律"(discipline)
Skill: "当用户问天气时,先查城市→再查天气→用友好格式回复→
        如果查不到就推荐最近的城市→回复要包含穿衣建议"

// MCP = 给你一把刀(工具)

工具调用失败:降级策略

// 生产环境中,工具调用会失败。必须有降级方案:

策略 1: 重试(Retry)
  tool_call → 失败 → 等 1s → 重试 → 成功
  适合:网络抖动、临时错误

策略 2: 备用工具(Fallback Tool)
  weather_api_A → 失败 → 尝试 weather_api_B
  适合:有多个同类工具可选

策略 3: 优雅降级(Graceful Degradation)
  精确天气 → 失败 → 返回"近期北京天气一般在 15-25°C"
  适合:有历史数据或通用知识可用

策略 4: 跳过并告知(Skip & Inform)
  "抱歉,天气查询暂时不可用,我先帮你完成其他部分"
  适合:非关键步骤

策略 5: 人工介入(Human-in-the-Loop)
  "这步需要你手动确认..."
  适合:高风险操作

结构化输出

// 问题:LLM 输出的是自然语言,怎么得到结构化 JSON?

方法 1: Prompt 引导(最简单)

方法 3: Structured Output(最可靠)
  response_format = {
    "type": "json_schema",
    "json_schema": { ... 严格的 schema 定义 }
  }
  ✅ 保证输出严格匹配指定的 JSON Schema

Q: MCP 是什么?本地工具、MCP Tools、Skill 三者的区别?

标准答案:MCP(Model Context Protocol)是 Anthropic 提出的开放标准,标准化 AI 应用连接外部工具/数据的方式,采用 JSON-RPC 协议。三者区别:

记忆的更新与遗忘

// 面试高频:传统记忆如何更新和遗忘?

更新策略:
1 Append     - 新信息追加(最简单)
2 Overwrite  - 新信息覆盖旧信息(如用户改了手机号)
3 Merge      - 新旧信息合并(冲突时保留最新)
4 Summarize  - 定期把零散记忆总结为精简版

遗忘策略:

2 TTL(Time To Live)      - 超过时间自动过期
3 Relevance               - 与当前任务无关的逐渐衰减
4 Compaction              - 压缩旧记忆为摘要

// OpenClaw 的做法:
// 短期记忆 = session transcript (JSONL)
// 长期记忆 = MEMORY.md + memory/YYYY-MM-DD.md
// 遗忘 = auto-compaction(自动摘要旧对话)

LRU 原理与实现

// LRU = Least Recently Used(最近最少使用)
// 用于限制记忆容量:满了就淘汰最久没用的

class LRUCache {
  constructor(capacity) {
    this.capacity = capacity;
    this.map = new Map();  // Map 保持插入顺序
  }

  get(key) {
    if (!this.map.has(key)) return undefined;
    const value = this.map.get(key);
    this.map.delete(key);     // 删除
    this.map.set(key, value);  // 重新插入(变成最新)
    return value;
  }

  put(key, value) {
    if (this.map.has(key)) this.map.delete(key);
    this.map.set(key, value);
    if (this.map.size > this.capacity) {
      // 淘汰最老的(Map 的第一个)
      const oldest = this.map.keys().next().value;
      this.map.delete(oldest);
    }
  }
}

向量数据库:Embedding 的家

上面我们把文本变成了向量(Embedding),也知道了怎么计算相似度。但实际场景中,你可能有几百万甚至上亿个文档向量。每来一个查询,都和所有向量逐一比较(暴力搜索)?一个 1536 维的向量比较一次就要 1536 次乘法--乘以一百万个文档,每次查询要十几亿次运算。太慢了!

类比:在图书馆找书。如果书随便堆在地上(暴力搜索),你得翻遍每一本。但如果书按主题分类、放在标了标签的书架上(索引),你先走到对的区域,再扫一遍那个书架就行了。向量数据库就是帮你"整理书架"的系统。

为什么不用传统数据库?

MySQL、PostgreSQL 这类关系型数据库擅长精确匹配:WHERE id = 123、WHERE name = '张三'。它们的索引(B-Tree)是为"等于""大于""小于"这类操作设计的。

但向量检索需要的是"在 1536 维空间中,找到离这个点最近的 10 个点"--这是一个完全不同的数学问题。传统索引结构在高维空间中效率极差(这叫"维度诅咒")。所以需要专门的向量数据库。

ANN(Approximate Nearest Neighbor,近似最近邻)

核心思路:用一点点精度换巨大的速度提升。不追求找到"最最最近"的向量,而是找到"非常非常近"的向量。实践中,ANN 通常能达到 99%+ 的召回率,但速度比暴力搜索快 100 倍以上。

HNSW(Hierarchical Navigable Small World)--最流行的 ANN 算法

类比:"六度分隔"理论。在社交网络里,任何两个人之间平均只隔 6 个人。HNSW 就是给向量建一个类似的"社交网络"--每个向量和一些"邻居"建立连接,搜索时沿着连接跳转,很快就能找到目标附近。

  HNSW 多层图结构:

  Layer 2 (稀疏)    A ──────────────── D        ← 高速公路:少数节点,远距离跳转
                    │                  │
  Layer 1 (中等)    A ─── B ────── D ── E       ← 主干道:更多节点
                    │     │        │    │
  Layer 0 (稠密)    A ─ B ─ C ─ D ─ E ─ F ─ G   ← 本地街道:所有节点,短距离连接

  搜索过程:从顶层开始 → 快速定位大致区域 → 逐层下沉 → 在底层精确搜索

Hybrid Search(混合检索)

// 问题:向量检索擅长语义匹配,但对精确关键词匹配弱
// BM25 擅长关键词匹配,但不懂语义

用户问:"OpenClaw 的 Gateway 端口是多少?"
  向量检索 → 可能返回"OpenClaw 架构介绍"(语义相关但没答案)
  BM25    → 可能返回"端口配置:18789"(精确匹配)

// Hybrid Search = 向量 + BM25 的融合
final_score = α × vector_score + (1-α) × bm25_score
  // α 通常取 0.5~0.7
  // 用 RRF (Reciprocal Rank Fusion) 融合两个排名列表效果更好

Rerank(重排序)

// 为什么需要 Rerank?
// 初始检索(embedding 向量匹配)是"粗筛"--快但不够精
// Rerank 是"精排"--慢但更准

初始检索(embedding)→ Top 20 候选
        ↓
Rerank 模型(cross-encoder)→ 重新打分排序
        ↓
取 Top 3-5 → 送给 LLM 生成

// 为什么 Rerank 比 embedding 更准?
Embedding(Bi-Encoder):  问题和文档分别编码,用余弦比较
  快(可以预计算),但精度有限

Rerank(Cross-Encoder):  问题+文档拼在一起,同时编码
  慢(不能预计算),但精度更高
  因为它能看到问题和文档之间的细粒度交互

// 常用 Rerank 模型:
// Cohere Rerank, bge-reranker-v2, Jina Reranker

RAG 评估指标

指标评估什么怎么算
Recall@K检索召回率Top-K 结果中包含正确答案的比例

RAGAS(最流行的 RAG 评估框架)- 自动计算 Faithfulness、Answer Relevance、Context Relevance

Q: 一个标准的 RAG 系统包含哪些模块?完整流程是什么?

标准答案:分两个阶段--

Q: Chunk size 怎么选?为什么很重要?

标准答案:Chunk size 直接影响检索质量。太小(如 100 token):检索精准但缺上下文,LLM 没有足够信息生成好的回答。太大(如 2000 token):上下文完整但检索不精准,且占用更多 context window。通常 256-512 token 是好的起点,需要根据具体数据和任务调优。

常见踩坑:❌ "RAG 比 grep 更高级所以应该用 RAG"--选工具要看场景,精确匹配场景 grep 更优。

08 · Harness Engineering + 幻觉

TL;DR:Harness 是"驾驭" Agent 的三大支柱--上下文工程、架构约束、熵管理。幻觉(Hallucination)是 Agent 最大的可靠性威胁。

三大支柱

1. 上下文工程(Context Engineering)
   控制 Agent 看到什么
   → System Prompt 设计、Skills 按需加载、Memory 管理

2. 架构约束(Architecture Constraints)
   控制 Agent 能做什么
   → 工具白名单、权限控制、人工确认节点

3. 熵管理(Entropy Management)
   控制混乱度
   → 日志记录、异常检测、自动熔断

🔥 幻觉(Hallucination)详解

// 幻觉 = LLM 生成了看起来正确但实际错误的内容
// 研究表明:领先 AI 模型在科学摘要中幻觉率高达 31%

三大类型(最新分类法):
1 事实型幻觉(Factual / Knowledge-based)
   → 事实错误:"爱因斯坦于 1950 年获诺贝尔奖"(错,是 1921)
   → 编造不存在的日期、数字、事件
   → 原因:训练数据缺失或矛盾

2 逻辑型幻觉(Logic-based)
   → 推理过程看起来对但结论错
   → 多步推理中错误累积
   → 原因:自回归模型的"模式延续"倾向

3 引用型幻觉(Attribution / Citation-based)
   → 引用虚构不存在的论文、链接、书籍
   → 格式看起来非常正规(作者名、年份、期刊名),但完全是编的
   → 原因:模型学到了"引用格式"的模式,但没有真实知识支撑

根本成因(从架构角度):
1 自回归生成 - 模型预测下一个 token 时倾向于"自信补全"
2 知识缺口 - 对长尾知识或最新信息覆盖不足
3 模式驱动 - 生成了语法正确但事实错误的续写
4 Context 稀释 - 长上下文中 Attention 衰减(Lost in the Middle)
5 训练数据噪声 - 互联网数据本身包含错误信息

7 层缓解策略(从轻到重):
1 Prompt 工程     - 明确指令:"只基于给定信息回答,不确定则说不知道"
2 RAG             - 检索真实文档作为上下文(最有效的单一策略)
3 Grounding       - 要求模型引用来源,每句话标注出处
4 Self-check      - 生成后让模型自己验证("请检查上述回答是否有事实错误")
5 Constrained 输出 - 结构化输出 + 限制回答范围
6 外部验证       - 自动化 fact-checker 验证关键事实
7 Fine-tuning    - 用领域专家标注的数据微调模型

// 实际生产中通常组合使用:RAG + Grounding + Self-check
🎯 面试高频题
Q: 什么是幻觉(Hallucination)?为什么会发生?如何减少?

标准答案:幻觉是 LLM 生成看似正确但实际错误的内容。成因:1训练数据噪声2模型倾向于补全而非拒绝3上下文信息不足。缓解:1RAG(提供真实参考文档)2要求引用来源3Self-consistency check4降低 temperature5结构化输出约束。

加分回答:区分三种幻觉类型(事实型/逻辑型/引用型),并提到 RAG 中的 Faithfulness 指标--评估生成内容是否忠于检索文档。引用型幻觉尤其危险--格式正规但完全虚构,读者很难识别。

常见踩坑:❌ "用 RAG 就能完全消除幻觉"--不能,但能大幅减少;RAG 本身也可能检索到错误文档。

手写 RAG Pipeline(Python)

from openai import OpenAI
import numpy as np

client = OpenAI()

# ── 1. 准备文档并切分 ──
documents = [
    "OpenClaw 的 Gateway 默认端口是 18789",
    "OpenClaw 用 WebSocket 协议通信,不是 HTTP",
    "Agent 的 System Prompt 包含 Tools、Skills、Bootstrap 文件",
    "Session 会在每天凌晨 4 点自动重置",
]

# ── 2. 文本 → 向量 ──
def embed(texts):
    resp = client.embeddings.create(model="text-embedding-3-small", input=texts)
    return [r.embedding for r in resp.data]

doc_vectors = embed(documents)

# ── 3. 检索 ──
def search(query, top_k=2):
    q_vec = embed([query])[0]
    scores = [np.dot(q_vec, dv) for dv in doc_vectors]
    top_idx = np.argsort(scores)[-top_k:][::-1]
    return [documents[i] for i in top_idx]

# ── 4. RAG 生成 ──
question = "OpenClaw 用什么协议?"
context = search(question)

answer = client.chat.completions.create(
    model="gpt-4",
    messages=[
        {"role": "system", "content": f"基于以下参考文档回答:\n{chr(10).join(context)}"},
        {"role": "user", "content": question}
    ]
)
print(answer.choices[0].message.content)
# → "OpenClaw 使用 WebSocket 协议通信"

// 前端 JavaScript:
const source = new EventSource("/api/chat");
source.onmessage = (event) => {
    const data = JSON.parse(event.data);
    if (data.type === "text_delta") {
        appendToChat(data.content);  // 逐字追加
    } else if (data.type === "tool_call") {
        showToolIndicator(data.name); // 显示"正在调用工具..."
    } else if (data.type === "done") {
        source.close();
    }
};

// 后端 Node.js:
app.get("/api/chat", (req, res) => {
    res.setHeader("Content-Type", "text/event-stream");
    res.setHeader("Cache-Control", "no-cache");

        model: "gpt-4", messages, stream: true
    });

    for await (const chunk of stream) {
        const delta = chunk.choices[0].delta;
        res.write(`data: ${JSON.stringify(delta)}\n\n`);
    }
    res.write("data: [DONE]\n\n");
    res.end();
});

训练基础:SFT / RLHF / DPO / GRPO

// 面试中经常问训练流程,不需要太深但要清楚区别

SFT(Supervised Fine-Tuning,监督微调)
  数据:(instruction, response) 对
  方法:标准交叉熵损失,教模型"照着好答案学"
  类比:老师给标准答案,学生照着抄

RLHF(Reinforcement Learning from Human Feedback)
  Step 1: 训练一个 Reward Model(人类偏好打分器)
  Step 2: 用 PPO 算法优化 LLM,最大化 Reward
  类比:先教裁判什么是好答案,再让学生根据裁判反馈改进

DPO(Direct Preference Optimization)
  数据:(prompt, good_response, bad_response) 三元组
  方法:直接优化模型偏好,不需要 Reward Model
  类比:直接告诉学生"A比B好",不需要裁判
  优点:比 RLHF 简单很多,训练更稳定

GRPO(Group Relative Policy Optimization)
  DeepSeek 提出,用于训练 DeepSeek-R1。核心创新:
  1 不需要 Reward Model--用可编程的奖励函数(如代码能不能跑通、数学答案对不对)
  2 组内相对排名--对同一个 prompt 生成一组回答,
     组内排名高的奖励、排名低的惩罚,不需要绝对分数
  3 训练流程(DeepSeek-R1):
     冷启动 RL(纯 GRPO,让模型自发学会思维链)
     → 收集 RL 产出的好样本做 SFT(固化格式)
     → 再做一轮 GRPO(进一步优化)
  类比:让一组学生各自做题,对答案后在组内排名,
     排名前几的奖励、排名后几的惩罚--不需要外部打分员
🎯 面试高频题
Q: PPO 和 DPO 在大模型对齐中的主要区别是什么?

标准答案:PPO 需要先训练一个 Reward Model,再用强化学习优化策略,流程复杂(2 阶段)。DPO 直接从人类偏好数据(good/bad 对比)优化模型,不需要 Reward Model(1 阶段),训练更简单稳定。DPO 的 loss 本质是让模型对好答案的概率增大,对差答案的概率减小。

加分回答:提到 GRPO(DeepSeek 提出),是 PPO 的简化版--不需要 Reward Model,用组内样本排名替代。

常见踩坑:❌ 说不出 DPO 的具体优势在哪--核心是"去掉了 Reward Model 这个中间环节"。
