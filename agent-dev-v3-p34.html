14 · Gateway 架构

TL;DR:Gateway 是 OpenClaw 的"神经中枢"--所有消息渠道、Agent、Node 都通过 WebSocket 连接到它。

核心架构

// 单进程长驻,默认端口 127.0.0.1:18789

               ┌─────── Gateway ────────┐
               │                        │
  WhatsApp ──→ │                        │ ←── macOS App
  Telegram ──→ │   WebSocket Server     │ ←── CLI
  Discord  ──→ │   (所有渠道/客户端      │ ←── WebChat
  飞书      ──→ │    都连到这里)          │ ←── Node (手机)
  Signal   ──→ │                        │
               └────────────────────────┘

// Wire Protocol(线路协议)
Request:  { type: "req",   id: "1", method: "agent", params: {...} }
Response: { type: "res",   id: "1", ok: true, payload: {...} }
Event:    { type: "event", event: "agent", payload: {...}, seq: 1 }

15 · Channel + Binding

TL;DR:Channel 是消息入口/出口,Binding 是路由规则。匹配优先级:peer > guildId > accountId > channel > default。

// 典型配置
bindings: [
  // 精确路由:特定人 → coding agent
  { agentId: "coding", match: { channel: "whatsapp",
    peer: { kind: "direct", id: "+8613800138000" } } },
  // 飞书全部 → sage
  { agentId: "sage", match: { channel: "feishu" } },
  // 兜底 → main
  { agentId: "main", match: { channel: "whatsapp" } },
]

16 · 多 Agent 路由(深入版)

TL;DR:每个 Agent 完全隔离(workspace + auth + sessions)。Sub-Agent 用 sessions_spawn 生成,promptMode: minimal 精简上下文。

// 每个 Agent 有三层隔离
Agent: sage
├── 🏠 Workspace (SOUL.md, AGENTS.md, USER.md, skills/)
├── 🔑 Auth     (auth-profiles.json - 独立凭证)
└── 💬 Sessions (sessions.json + *.jsonl - 独立对话)

// Sub-Agent 生命周期
sessions_spawn({ task: "...", mode: "run" })
  → 继承父 workspace → 独立 session → 完成后推送结果
  → promptMode: "minimal"(只注入 AGENTS.md + TOOLS.md)

17 · Plugin + Context Engine

TL;DR:Plugin 是 OpenClaw 的扩展机制(模型/渠道/工具),Context Engine 控制模型"看到什么"。

// Plugin 四层架构
1. Manifest + Discovery  → 发现候选插件
2. Enablement + Validation → 决定启用/禁用
3. Runtime Loading       → 调用 register(api)
4. Surface Consumption   → Gateway 各模块读取 Registry

// Context Engine 四阶段生命周期
ingest(message)          → 存储新消息
assemble(msgs, budget)   → 精选送进 context window 的消息 ⭐
compact(sessionId)       → 压缩旧历史
afterTurn(session)       → 后处理

18 · Skills 系统

TL;DR:Skill = Agent 的专业手册,按需加载不占 context。三级优先:Workspace > Managed > Bundled。

渐进式披露(Progressive Disclosure)

// 问题:Agent 有 50 个 Skill,每个 SKILL.md 几千字
// 全放进 context window → 几万 token → 爆了

// 解决:渐进式披露
Level 1: System Prompt 只包含目录
  "可用技能:weather(天气), github(GitHub)..."
  → 每个 Skill 只占 ~50 tokens

Level 2: 需要时读取 SKILL.md
  Agent → read("~/.openclaw/skills/weather/SKILL.md")
  → 只有用到时才加载完整指令

// 类似 MCP 的 Progressive Discovery(2026.01 新增)
// Skill 更早实现了这种"按需加载"的设计
🎯 面试高频题
Q: Skill 了解吗?渐进式披露是怎么实现的?Skill 里面有没有工具?

标准答案:Skill 是 Agent 的行为指令文件(SKILL.md),定义"遇到什么场景,怎么做"。渐进式披露:system prompt 只注入 Skill 目录(名称+描述+路径),Agent 需要时再用 read 工具加载完整内容。Skill 本身不是工具,但 Skill 的指令中通常会引导 Agent 调用特定工具(如 exec、web_search)。

加分回答:提到三级加载优先级(Workspace > Managed > Bundled)和 MCP 在 2026.01 也加入了类似的 Progressive Discovery(工具搜索),解决 context 污染问题。

常见踩坑:❌ "Skill 就是工具"--不是,Skill 是指令/方法论,工具是能力。

19 · Prompt Pipeline 全链路

TL;DR:用户消息到 Agent 回复经过:Channel接收 → Binding路由 → Session解析 → Queue排队 → System Prompt组装 → LLM推理 → Tool执行 → Reply投递。

Prompt Pipeline(简化版)
// System Prompt 由以下部分组装:
1 Tooling     - 可用工具列表
2 Safety      - 安全准则
3 Skills      - 技能目录(按需加载)
4 Workspace   - 工作目录路径
5 Date/Time   - 当前时间
6 Runtime     - 模型/OS/主机信息
7 Project Context(Bootstrap 文件):
   - AGENTS.md  - 操作指令
   - SOUL.md    - 人格定义  ← 你正在体验这个!
   - USER.md    - 用户信息
   - IDENTITY.md
   - TOOLS.md
   - MEMORY.md  - 长期记忆

// Sub-Agent 只注入 AGENTS.md + TOOLS.md(精简模式)

20 · OpenClaw 源码导读

TL;DR:核心调用链:agent RPC → agentCommand → runEmbeddedPiAgent → subscribeEmbeddedPiSession。理解这条链,就理解了 80% 的源码。

核心调用链

agent RPC(入口)
  → 验证参数,解析 session,返回 runId
       ↓
agentCommand
  → 解析模型 + 加载 Skills + 构建 pi session
       ↓
runEmbeddedPiAgent ⭐
  → per-session + global queue 序列化
  → 构建 pi session → 启动推理 → 超时管理
       ↓
subscribeEmbeddedPiSession
  → 桥接 pi-agent-core 事件到 OpenClaw 流
  → tool events → "tool" 流
  → assistant deltas → "assistant" 流
  → lifecycle → "lifecycle" 流 (start/end/error)

四大核心抽象

🔄 Agent Loop      = 发动机    - 驱动一切运转
🔌 Plugin Registry = 管线系统  - 连接所有组件
💬 Session Manager = 记录仪    - 管理所有会话
🧠 Context Engine  = 导航系统  - 决定模型看到什么

推荐阅读路径

Level 1: 读文档(你刚读完的这 20 章 ✅)
Level 2: 读配置(openclaw.json 的每个字段)
Level 3: 跟着一条消息走源码
  → 从 Channel 收到消息
  → 追踪 Binding → Session → Queue → Agent Loop
  → 看 System Prompt 怎么组装
  → 看 LLM 推理 + Tool 执行
  → 看 Reply 怎么投递回去
